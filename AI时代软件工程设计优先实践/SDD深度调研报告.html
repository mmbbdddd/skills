前置 1：无知声明

最不确定的三件事

  1. SDD + AI执行的长期维护收益有无超过18个月的实证数据？
    猜测：目前没有。大部分数据来自厂商自报短期实验（Google 12周、EPAM 6周），最长的第三方研究（CMU 2025）只追踪到8个月。Odessa论文（2026-02）提供了3年ROI模型，但是经济学推导而非实测。
  2. Vibe Coding的代价拐点具体在哪个时间点出现？
    猜测：约3-4个月。CMU数据显示AI速度增益在第3个月消失，METR显示经验丰富开发者甚至更慢，但不同项目规模和团队水平的拐点差异可能超过10倍。
  3. 中国市场的SDD实践是否和美国有本质差异？
    猜测：中国更偏向"实用主义混合"——用SDD框架管理架构边界但允许内部Vibe Coding，与美国"全栈规范驱动"理念存在张力。这来自已知案例（民生银行、平安科技），但缺乏跨市场对照实验。

1.3 交叉分析

内部张力：三个Top产品都宣称"SDD是未来"，但实现路径截然不同。Spec Kit是"无头CLI"（附着于任何AI助手），Kiro是"内置SDD的完整IDE"，Qoder是"双模式平台"（既支持Vibe又支持Spec）。这种差异反映了一个根本张力：SDD究竟是方法论（应附着于任何工具）还是产品（需深度整合到IDE架构）？

跨步矛盾：暂不评估（待后续步骤填充）。

论证链检验：场景分解中的速度需求打分依赖个人判断而非用户调研数据——这是论证链的薄弱环节。论证强度：中

步骤二：技术范式/原理/实现

2.1 范式识别

当前AI辅助软件工程中存在四个主要技术范式，它们的差异不在于"是否使用AI"，而在于人类意图和AI执行之间的接口是什么：

范式人-AI接口典型工具人类角色
Vibe Coding（氛围编程）自然语言对话Cursor Agent/Chat、ChatGPT Canvas、早期Copilot Chat需求描述者+调试者
Plan-First Coding（计划先行）自然语言计划文档(.plan.md)Cursor Plan Mode、Copilot Plan Mode、Windsurf Cascade计划审查者+代码审查者
Spec-Driven Development (SDD)（规格驱动）结构化/半结构化规格文件(.md)，分阶段审查Spec Kit、Kiro、Qoder Quest、Tessl Framework、CoStrict规格编写者+关卡审查者
Spec-as-Source（规格即源码）规格文件是唯一编辑对象，代码完全自动生成且不手动编辑Tessl（目标中，未达）、Martinelli方案纯规格编写者

2.2 范式一：Spec-Driven Development (SDD) — 深度拆解（R18）

(a) 组件清单

组件功能输入输出是否可替换替代方案依赖关系
Constitution（宪章）项目不变量：编码标准、安全规则、架构原则团队决策constitution.md否（核心）无替代——这是SDD的"DNA"被Specify/Plan/Tasks引用
Specify（规格化）将需求转化为结构化User Story + 验收标准用户需求（自然语言）spec.md（含User Journey、Goals、Acceptance Criteria）否BDD Gherkin、EARS语法依赖Constitution；被Plan消费
Clarify（澄清）AI主动提问消除规格歧义spec.md更新后的spec.md + Q&A日志是人工审查替代依赖Specify
Plan（技术规划）技术栈选择、架构设计、依赖分析spec.md + constitution.mdplan.md（含架构图、技术决策、trade-off分析）是手动ADR（Architecture Decision Record）依赖Specify；被Tasks消费
Tasks（任务分解）将Plan分解为1-2文件的原子任务plan.mdtasks.md（含checklist、依赖关系）否手动拆分+JIRA依赖Plan；被Implement消费
Implement（执行）AI按tasks.md逐任务生成代码tasks.md + 代码库代码变更（PR）是手动编码+AI辅助审查依赖Tasks
Review Gate（审查关卡）人类在每个阶段审批或退回阶段产物批准/退回+修改意见否无——这是SDD的"人环路"核心所有阶段间
Agent Hooks（自动化钩子）文件变更触发自动动作（lint、test、docs更新）文件变更事件自动执行的脚本/命令是Git hooks、CI/CD pipeline依赖Implement

(b) 运行步骤/流程（以Spec Kit为例，含瓶颈标注）

  1. 初始化项目 → 输入：项目名和AI助手选择 → 处理：创建目录结构、.speckit/配置 → 输出：空项目骨架 → 瓶颈：如果AI助手未配置完善，此步即失败
  2. 编写Constitution → 输入：团队编码规范、安全策略 → 处理：AI辅助起草，人工审查2-3天 → 输出：constitution.md → 瓶颈：这是质量天花板——弱宪章→全流程低质量
  3. 撰写Spec → 输入：业务需求 → 处理：AI生成User Story + 验收标准 → 输出：spec.md → 决策点：人类审查通过？→否则返回修改
  4. AI澄清 → 输入：spec.md → 处理：AI扫描歧义、缺失边界条件 → 输出：Q&A列表 → 决策点：人类回答所有问题→spec更新
  5. 技术规划 → 输入：spec.md + constitution.md → 处理：AI推荐技术栈、画出架构 → 输出：plan.md → 决策点：人类批准技术选择？
  6. 任务分解 → 输入：plan.md → 处理：AI拆解为1-2文件/任务的原子单元 → 输出：tasks.md checklist → 瓶颈：分解粒度错误会导致重复工作或漏项
  7. 逐任务执行 → 输入：单个task + 代码库 → 处理：AI生成代码 → 输出：代码diff → 决策点：每任务完成后人类审查
  8. 自动化验证 → 输入：代码变更 → 处理：Agent Hooks触发lint/test/build → 输出：通过/失败报告 → 失败则AI自动修复或人类介入
  9. 集成测试 → 输入：所有任务完成 → 处理：端到端测试 → 输出：测试报告 → 瓶颈：AI擅单任务不擅跨任务集成——此处失败率最高
  10. 文档归档 → 输入：全部阶段产物 → 处理：生成最终文档 → 输出：可审计的完整traceability链 → 轻量步骤

2.4 交叉分析

3.3 路径依赖分析

当前SDD的形态高度偶然。几个关键节点如果不同，整个行业走向可能完全不同：

  • 如果Karpathy创造的不是"Vibe Coding"而是"Spec Coding"——2024年初大语言模型的能力恰好支持对话式编码但不足以自主执行复杂多文件任务。Vibe Coding的流行部分是因为"当时的AI只能这么用"。如果Karpathy在2025年底（模型能力更强时）才创造这个词，Vibe Coding的定义和实践可能完全不同。
  • 如果OpenAI收购了Windsurf——SDD可能被推迟数年。OpenAI的产品哲学是"降低使用门槛"而非"增加结构化流程"。一个OpenAI拥有的Windsurf很可能会向Vibe Coding方向深度优化。
  • 如果GitHub没有开源Spec Kit——SDD可能只是AWS(Kiro)和阿里(Qoder)的闭源特性，缺乏一个"中立标准"。AGPL开源让SDD方法论可以脱离具体IDE传播，这是关键的历史偶然。

步骤五：泛化分析

5.1 范式外推

如果将SDD的思想外推到极致，软件开发会变成什么？

  • Spec-as-Source极限（Tessl方向）：开发者从来不看代码。规格是唯一的编辑对象。代码是纯粹的下游产物——像编译器输出的汇编代码一样，不应被人手修改。这个极限的问题在于：当代码出Bug时，你是修Bug还是修规格？如果修代码，规格就过时了。如果修规格后重新生成，可能引入新的不相关变化。非确定性的LLM使"确定性重新生成"不可能。
  • CI/CD即编译器：将CI/CD管道中的门禁（lint、SAST、覆盖率、架构合规）从"质量检查"升级为"质量编译"——不通过门禁=构建失败。这实际上把软件质量规则从"建议"变成了"物理定律"。CircleCI Chunk和Parasoft 2025.2是这个方向的早期信号。
  • Architecture-as-Code极限：将架构规则（如"服务不能向上调用依赖层级"）以可执行代码编码，让AI生成的代码如果违反架构规则则自动被CI/CD拒绝。NDepend、OPA/Conftest、ArchUnit是这个方向的现有工具。

5.2 沉默证据

  • 放弃SDD的团队：有多少团队试了SDD后放弃了？我们没有数据，因为放弃者不写博客。我们只能看到坚持者和倡导者的声音。Scott Logic是罕见的公开"放弃"（至少是负面评价），这本身是沉默证据冰山的一角。

步骤七：红队反证时刻（R6 · 强制）

攻击#1：「SDD可能比Vibe Coding产生更多Bug」

攻击目标：SDD的核心价值主张——"结构化规格减少AI幻觉和Bug"。

攻击内容：Scott Logic实验中SDD生成的1,000 LOC代码包含一个明显的Bug——规格再完善也没有阻止它。EPAM实验中AI Agent"倾向于写而非读"，生成了重复服务。更根本的质疑：规约中的Bug比代码中的Bug更难发现——因为规格使用自然语言，没有编译器/测试来验证它的正确性。代码中的Bug可以被静态分析工具捕获（如GitClear警告增加）；规格中的逻辑错误只能通过人类审查发现，而人类审查规格的效率远低于审查代码。

证据：Scott Logic原文："The SDD-generated code still had a simple, obvious bug that wouldn't be caught by improving the spec." [来源]。CodeRabbit发现AI生成代码的"Logic & Correctness"问题比人类代码高75%。

严重程度：颠覆——如果规格中的Bug比代码中的Bug更难发现且一样多，那么SDD只是改变了Bug的存在形式而非减少Bug。

攻击#2：「当前AI模型能力不足以支撑SDD的全部承诺」

攻击目标：SDD工具的"AI自主执行"能力。

攻击内容：多文件任务的Pass@1只有19.36%（学术研究），而SDD的核心操作就是多文件/多任务执行。Kiro用户反馈"长对话后AI忘记之前构建的方案(Doom Prompting)"。METR研究显示AI在大型/复杂代码库中反而让经验开发者变慢19%。这些数据共同指向一个结论：SDD的全部承诺（spec→完整可工作的多文件实现）需要的AI能力目前还不稳定。SDD工具目前更像是"AI辅助的瀑布模型"——人类仍然需要介入到每个阶段来修正AI的输出。

证据：METR RCT 19%更慢[来源]。多文件Pass@1=19.36%（学术研究，引自步骤二数据）。Kiro Doom Prompting来自GitHub Issue #1665[来源]。

严重程度：削弱——不否定SDD的方向，但说明当前实施受限于模型能力，短期期望需要下调。

攻击#3：「SDD可能是瀑布模型的AI复兴——而瀑布模型失败的原因并未消失」

攻击目标：SDD作为方法论本身的合理性。

攻击内容：SDD的核心假设是"前期规格投入可以在后期获得回报"。这与瀑布模型的核心假设相同。瀑布模型被敏捷运动推翻的原因是：需求变化——前期详细规格在编写完毕时已经部分过时。AI没有改变"需求会变化"这个基本事实。如果业务需求在SDD的6阶段流程中发生变化（这在多数商业环境中是常态），返工成本包括：修改Constitution + 修改Spec + 修改Plan + 重新生成Tasks + 重新AI执行 + 重新人类审查——这个返工循环比直接手动修改Vibe Coding生成的代码（或直接用AI重新生成）更昂贵。

证据：Martin Fowler博客（Birgitta Böckeler）："SDD的工作流仍然是精细且观念化的……我们可能在重新学习一个痛苦的教训：手动为AI编写详细规则最终不会扩展（doesn't scale）[来源]"。MDD（Model-Driven Development）的商业失败——UML工具从统治地位到几乎消失——是行业前车之鉴。

严重程度：颠覆——如果瀑布模型的核心问题（需求变化成本）在SDD中同样存在且可能更严重，那么SDD的适用场景被严重限缩到"需求极其稳定"的领域（金融核心系统、安全关键系统等），而非SDD倡导者宣称的"通用方法论"。

攻击#4（框架层）：「整个分析框架假设'代码质量'可以用churn率、复杂度、静态分析警告和Bug数量来衡量——但SDD可能恰恰破坏了这些指标的效度」

攻击目标：本报告的分析框架本身。

攻击内容：SDD生成的代码倾向于过度工程化（Kiro的多余测试文件、冗余fallback）——这会降低churn率（因为代码太多没人敢动）和增加复杂度（因为过度工程化）。在传统质量指标上，一个过度工程化的代码库可能看起来"更稳定"（低churn）和"更复杂"（高复杂度）——而这两个指标在传统框架中分别指向"好"（低churn=代码稳定）和"坏"（高复杂度=难以理解）。SDD可能创建一种新的"质量幻觉"：表面指标良好的代码库实际上过度僵硬、难以修改、充满间接层。我们的整个分析框架——包括GitClear的churn率、CMU的复杂度指标、SIG的可维护性评分——可能对SDD生成的代码类型产生系统性误判。

证据：Kiro用户反馈的过度工程化倾向[来源]。MIT Armando Solar-Lezama："AI像一张全新的信用卡，允许我们以前所未有的方式积累技术债[来源]"——这意味着传统指标可能低估了AI引入的新型债务。

严重程度：颠覆——不针对任何具体结论，而是攻击了支撑所有结论的衡量标准本身。这迫使我们在所有结论上附加一个元级别的不确定性：我们目前的测量工具可能无法正确评估SDD生成代码的真实长期质量。

红队总结

四条攻击中两条被评定为"颠覆"级别。核心风险集中在：(a) SDD可能只是将Bug从代码层移到规格层（更难检测）；(b) SDD面临与瀑布模型相同的"需求变化成本"问题；(c) 我们的衡量工具本身可能对SDD产生的代码类型产生系统性误判。这些不是边缘风险——它们攻击的是SDD的核心价值主张和评估框架。

步骤八：结论 + 行动建议

8.1 结论分层输出

🔴 立即行动（1-2条）

  1. 不要将SDD作为所有项目的默认方法论。
    判断：当前SDD工具全部未达GA成熟度（Spec Kit v0.0.72、Kiro公测、Qoder公测），且小型项目净收益为负的结论已被多源确认。在以下条件下不选SDD：项目总规模<5K LOC、需求频繁变化（<2周迭代）、个人项目、原型验证阶段。在以下条件下考虑SDD：金融/合规/安全关键系统、多人协作(>3人)的中大型项目(>50K LOC)、需求稳定周期>1个月的场景。天花板：当前工具链不成熟（碎片化、无跨工具规格标准）会限制SDD的最大效率——预估12-18个月后工具成熟度达到可接受水平。更换信号：当某个工具达到GA状态+出现独立长期RCT证实SDD维护收益时，可以更积极地推广。
    本结论最可能出错的原因是：模型能力快速提升（6-12个月内）可能使SDD的"AI能力瓶颈"被大幅缓解，从而改变SDD的收益/成本比。
  2. 立即建立AI代码的CI/CD强制门禁——无论是否采用SDD。
    判断：已有多源确凿证据表明AI代码的静态分析警告+30%（CMU）、安全漏洞+76%（Apiiro）、缺陷密度+70%（CodeRabbit）。这些数据不受SDD是否有效的影响——即使SDD有效，你仍然需要门禁。最小可行的门禁：SAST(CodeQL/Semgrep) + SCA(依赖扫描) + 测试覆盖率阈值 + AI辅助标记的PR必须高级审查者批准。不要选"等SDD工具成熟后再做门禁"——门禁是今天就能部署的防御层，与SDD独立且互补。天花板：门禁只能防御已知模式——新型AI代码缺陷（如过度工程化、架构漂移）不一定被现有门禁捕获。更换信号：当AI代码的缺陷密度降至与人类代码无显著差异时，可以放松AI代码的特殊门禁要求。
    本结论最可能出错的原因是：如果组织对AI代码的审查流程过于严格导致开发速度锐减（"门禁瓶颈"取代"代码生成瓶颈"），整体效能可能不升反降。需要在部署时做好门禁延迟度量。

🟡 趋势观察（1-2条）

  1. SDD规格标准化战争即将开始——关注结果但不急于站队。
    判断：Spec Kit、Kiro、Qoder、Tessl的规格格式互不兼容——这是行业碎片化的经典早期信号。历史表明这种战争通常有两种结果：(a) 一个开源标准胜出（类似OpenAPI for REST），(b) 持续碎片化导致各自生态封闭（类似早期的移动应用平台）。如果(a)发生，SDD的工具切换成本将从"高"降至"中等"——这将消除SDD采纳的一个主要障碍。如果(b)发生，企业将被迫在工具层面"选边"。当前信号偏向(a)——因为Spec Kit是AGPL开源且GitHub拥有最大的开发者网络效应。行动：在6-12个月内观察是否有跨工具规格格式标准化倡议出现，不急于在工具锁定上押注。
    本结论最可能出错的原因是：GitHub可能改变Spec Kit的许可或方向（被微软商业战略影响），从而消除开源标准化窗口。历史上有太多企业赞助的"开源标准"最终变成特洛伊木马。
  2. 中国市场将在SDD的"混合模式"路径上独立演进——关注对中国团队更适用的工具。
    判断：H3已被确认——中国市场倾向"SDD管架构限制+Vibe Coding做内部实现"的混合策略（民生银行、平安科技）。这说明中国团队不会简单跟随美国的"纯SDD"路线，而是发展出适合中国快速迭代文化+金融监管要求的本土实践。深信服CoStrict（私有化部署+严肃编程）和Qoder（Quest异步模式）是目前最值得关注的中国SDD工具。CodeBuddy的全流程平台提供了另一种可能性——将SDD集成到从产品到部署的完整链路中而非仅代码层。
    本结论最可能出错的原因是：中国AI编程市场受模型能力限制（中文LLM相对于GPT/Claude的差距），如果国内模型能力在2026年显著突破，中国工具可能更快地追赶上纯SDD路线。

🟢 风险警示（1-2条）

  1. "空心高级工程师"现象是真实且正在加速的风险——与是否采用SDD无关。
    判断：Zenodo 2025系统分析+Hacker News广泛讨论+DORA"S形曲线"人才分布观察——多源确认此现象真实存在且正在恶化。SDD可能部分缓解（因为强制人类参与审查环节），但不能根除——审查能力本身需要深度技术理解。风险表现：初级工程师在1-2年内通过AI获得"10倍速度"，被提拔为高级工程师，但在AI宕机或遇到AI无法处理的边缘案例时完全无能。组织级后果：当"空心高级工程师"在组织中达到临界质量（估计30-40%），整个团队的故障恢复能力将断崖式下降。微软/Azure 2025年10月Copilot宕机事件是预警信号（依赖链单一故障点导致大面积开发停滞）。缓解措施：(a) 定期"无AI日"——强制无AI辅助编程练习；(b) 反向面试——评估候选人在无AI工具时解决基础问题的能力；(c) "考古学冲刺"——定期让团队逆向工程并文档化AI生成的代码段。
    本结论最可能出错的原因是：如果AI能力在2027年前达到"几乎不出错"的水平，"空心高级"可能不是问题而是适应——就像现在没有人因为不会手算开方根而被视为不合格的数学家。
  2. 规格≠代码——SDD当前存在"规格幻觉"风险，需严格验证。
    判断：红队攻击#1和#4揭示了核心问题：规格的正确性无法被机器验证（不像代码有编译器和测试）。在SDD中，规格成为新的"隐藏层Bug源"。CodeRabbit和Scott Logic的数据暗示：AI生成的规格可能包含与AI生成的代码同等频率的逻辑错误，但这些错误在规格阶段更难被发现（因为审查者倾向于关注"需求是否完整"而非"需求是否逻辑自洽"）。缓解措施：(a) 对关键规格进行可执行化（将书面验收标准转化为自动化验收测试——BDD风格）；(b) 两人规则——所有规格必须由第二个人类审查而非仅AI+单个开发者；(c) 定期将规格翻译为形式化约束（Pydantic/Zod schema），利用类型系统捕获规格中的矛盾。
    本结论最可能出错的原因是：如果未来AI模型的推理能力足够强（接近人类水平），AI可能比人类更擅长发现规格中的逻辑矛盾——届时"规格幻觉"从风险变为非问题。

8.2 回头对无知声明

之前不确定的现在有答案了吗
SDD是否有超过18个月的长期维护实证数据？ 没有。最长的第三方研究是CMU的8个月（2024-01 ~ 2025-08），最长的经济模型是Odessa论文的3年ROI（但非实测）。行业目前处于"信仰飞跃"阶段——推断长期收益但未经长期数据验证。
Vibe Coding的代价拐点具体在哪里？ 部分回答了。CMU数据显示AI速度增益在第3个月消失，代码质量指标在采纳后持续恶化而非收敛。但"拐点"因项目规模、团队经验和约束严格程度差异可能超过10倍——没有一个通用数字。
中国和美国SDD实践是否有本质差异？ 有，答案是"混合模式程度"的差异。中国的"实用主义混合"更普遍（SDD管架构+内部Vibe Coding）。

8.3 证据脆弱性 Top 3（R14）

  1. "SDD降低长期维护成本"——基于经济学模型（Odessa论文）+短期实验推断，缺乏独立长期RCT。如果2027年出现第一个12个月以上的RCT且结果为中性或负面，这将动摇整个SDD运动的理论基础。
  2. "中国vs美国SDD市场差异"——仅基于4个中国案例（民生银行、平安科技、腾讯、阿里）vs 4个美国案例（Spec Kit、Kiro、Tessl、Cursor），样本量小且案例选择偏向"已在公开场合宣传SDD的组织"——存在自我选择偏差。
  3. GitClear 211M LOC数据的"AI归因"——GitClear分析的是代码变化趋势（重构减少、churn增加），但这些变化与AI使用的因果关系并非直接测量——相关性不等于因果性。有可能2021-2024年间开发者实践的整体变化（更快的发布节奏、更少的文档、更多的微服务等）独立于AI使用就导致了这些趋势。

认知局限（R13）

  1. WebFetch不可用导致的二手信息偏差：本报告所有数据均通过WebSearch的搜索结果摘要获取，无法直接从原始页面验证完整上下文。搜索引擎可能对高流量/正面内容有排名偏好，这意味着本报告的"信息池"可能存在系统性乐观偏差。
  2. 发表偏差（Publication Bias）：几乎所有公开可查的SDD案例都来自已经在使用并愿意宣传SDD的组织——我们完全听不到"试用后放弃"的团队声音。这个偏差可能导致SDD的正面效果被严重高估。Scott Logic的负面评测是罕见且珍贵的反例——它本身的存在暗示更多类似负面反馈可能未被发表。
  3. 时间窗口限制：整个SDD工具链的历史不到12个月（Spec Kit 2025-05发布）。在这么短的时间窗口内，任何关于"长期维护成本"的结论在本质上都是推测。
  4. 英中信息不对称：英文搜索产生的结果数量和质量远高于中文搜索——这意味着中国市场的实践可能未被充分代表，尽管我们尽力搜集了中文来源。
  5. 模型能力变化速度 > 研究发表速度：本报告中引用的基准测试（如METR的2025年早期实验、CMU的Cursor数据集）可能已被更新的模型能力超越。这是AI领域研究的结构性挑战——任何超过3个月的benchmark在极限意义上已经过时。
  6. GDPR/隐私限制：最能揭示AI代码真实质量的企业内部数据（如GitHub Copilot在Google/Meta的内部采纳率和回滚率）对研究者不可见。
