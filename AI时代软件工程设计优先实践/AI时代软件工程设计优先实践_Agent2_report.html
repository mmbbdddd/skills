1. 无知声明

对调研对象最不确定的三件事：

  1. SDD的"隐性成本"规模有多大？猜测：规格编写+维护的时间可能吃掉了50%以上的效率提升，但缺乏系统性的成本核算数据来验证。
  2. SDD是否在大型组织（1000+工程师）中实际可规模化？猜测：目前所有"成功案例"（Salesforce、阿里5人7天、Bun）都是高度自选的——可能是幸存者偏差很重的样本。
  3. 中国市场的SDD落地是否独立于全球路径？猜测：由于模型可用性差异（通义千问 vs Claude）、合规需求（金融行业自主可控）、以及SDD概念引入的时间差，中国市场可能演化出了不同于全球的实践路径。

没条件查到的信息：

  • 企业内部SDD实践的详细AB测试数据（公司视为商业机密）
  • AI编码工具的真实token消耗/成本明细（厂商不愿公开）
  • SDD失败案例的内部复盘（组织通常不会主动分享失败）
  • 中国市场企业级SDD工具的私有化部署数据

1.1 核心产品深度分析（Top 3）

A. GitHub Spec Kit

维度详情
产品形态开源CLI工具（MIT许可，Python 3.11+，uv包管理器），提供SDD工作流的全套命令（/speckit.constitution → specify → plan → clarify → tasks → analyze → implement）
面向谁使用AI编码代理的开发团队；支持30+代理集成（Copilot、Claude Code、Cursor、Gemini CLI、Windsurf等）
核心交互开发者通过slash命令在终端中触发SDD各阶段；AI代理据此生成Markdown规格文件，再转化为任务分解和代码实现
GitHub Stars~115,000（2026-06-29）
社区规模200+贡献者，105个社区扩展，22个预设（Spring Boot、Go微服务等）
活跃地理市场全球（美国、欧洲、中国均有采用案例）；中国用户常将其与通义千问结合使用

(a) 独立来源评价：

  • Birgitta Böckeler（ThoughtWorks）在Martin Fowler网站上的深度评测（2025-10-15）指出：Spec Kit生成的Markdown文件"极其冗长、重复、审查起来令人厌倦"，有时候"我宁愿直接审查代码而不是这些Markdown文件"。她还发现Agents频繁忽略checklist项目或"过度热情地遵循规则"。
  • Scott Logic（2025-11-26）的实测结论："Spec Kit把你拖回了过去"——认为它是重新发明的瀑布模型，指出"没有SDD我的效率大约高出10倍"。

(b) 真实使用案例：

  • 阿里巴巴淘特导购团队将Spec Kit与通义千问（qwen3 coder plus）结合使用，技术方案填写从2小时降至20分钟，代码实现从1天降至2小时。（来源：阿里云开发者社区，2026-06-29检索）
  • 民生银行将其作为SDD工具链的一部分，用于私域CodeAgent研发探索。（来源：SegmentFault，2026-06-29检索）

(c) 已知局限/失败案例：

  • 小bug修复场景下产生4个用户故事+16个验收标准——"用大炮打蚊子"（Böckeler）
  • INNOQ实测：简单"创建用户资料"膨胀为1,070行Markdown+19个验收标准
  • Constitution机制被批评为"加载骰子"——依赖引入原则使AI总是选择零依赖方案而非最佳方案（alexcpn/speckit_test分析）

B. Amazon Kiro

维度详情
产品形态基于Code OSS的独立Agentic IDE，内置Claude模型（Sonnet 3.7/4.5、Haiku 4.5）。2025-07-15发布预览，2025-11-18 GA。
面向谁个人开发者和团队（有Team Plan支持IAM Identity Center统一管理）
核心交互四步工作流：输入需求→生成用户故事（EARS格式）→生成技术设计文档→生成依赖排序任务；支持Hooks（文件保存/创建时自动触发代理）和Checkpointing（回退到任意执行状态）
定价Free（50交互/月）→ Pro $19（1,000）→ Pro+ $39（3,000）→ Pay-as-you-go $0.04/交互
活跃地理市场全球（AWS主导市场：北美、欧洲、亚太）；不需要AWS账号，可用GitHub/Google登录

(a) 独立来源评价：

  • Böckeler评价：Kiro的工作流对于小型任务"过于僵化"——小bug修复被转化为4个用户故事和16个验收标准。但她也认可Kiro的"steering"记忆银行概念有价值。
  • Forbes（2025-07-15）：强调Kiro区别于Cursor/Copilot的关键在于"structure, traceability, and production-readiness"而非快速原型。

(b) 真实使用案例：

  • Forbes报道提到Kiro维护"living specs"与代码同步，为企业（关注SOC2/HIPAA合规）提供审计追踪。（Forbes，2026-06-29检索）

(c) 已知局限：

  • 工作流刚性，对小型bug修复和探索性原型不友好
  • 暂时不支持社区扩展插件生态（与GitHub Spec Kit相比）
  • 经济性在下滑——AWS母公司2025-2026年面临AI成本压力，Kiro的长期定价策略不确定

C. Qoder（阿里巴巴）

维度详情
产品形态阿里云旗下的AI编程平台，核心是Quest 1.0自主智能体（2026年1月上线）。集成IDE+CLI+桌面端三大入口。
面向谁阿里生态内的开发者和企业客户；特别强调了"非程序员"用户的桌面应用场景（QoderWork）
核心交互四阶段SDD：Specify（人机共创规格）→ Plan（AI设计方案）→ Implement（AI+多Quest代理并行执行）→ Validate（AI自测+AI Code Review）
独有特性自学习与进化能力（学习代码风格/项目规范/领域知识）、多代理并行（Git Worktree隔离）、自动恢复（诊断并修复bug）、小时级长时间任务
活跃地理市场中国为主（依托阿里云生态和通义千问模型）；通过阿里云国际版向东南亚/中东扩展中

(a) 独立来源评价：

  • 雷峰网（2026-01）：报道Quest 1.0的"自学习与进化"能力，但未提供独立基准测试。
  • 品玩（2026-01）：强调Qoder的"软件意图化"范式转变——从Software Engineering到Software Intentioning。这些中文科技媒体倾向于正面报道阿里系产品，独立性有限。

(b) 真实使用案例：

  • 5人7天构建QoderWork桌面应用的案例（官方自述，Qoder官方博客）。需注意：这是"喝自己的香槟"（dogfooding）——用Qoder构建Qoder自己的产品，这让案例的说服力打了折扣，因为团队对工具的熟悉程度远超外部用户。

(c) 已知局限：

  • 强依赖阿里云生态和通义千问模型；非阿里系用户面临供应商锁定风险
  • "自学习与进化"能力缺乏第三方验证
  • 企业级场景的规模化数据有限（vs GitHub Spec Kit的100K+星社区）
  • 国际市场接受度低——目前主要在中国、东南亚有存在

1.2 场景强制分解（R7）

≥3个差异显著的场景，需求特征按1-5分量化。

维度场景A：金融合规系统（民生银行案例）场景B：互联网快速迭代（阿里淘特案例）场景C：单人/小团队创业原型（YC Startup案例）

速度需求2 — 合规>速度5 — 快速上线是关键5 — 速度就是一切
成本敏感度3 — 有预算但有上限4 — 需要明确的ROI5 — 极低的资金消耗率
可追溯性需求5 — 监管要求完整审计2 — 基本追溯即可1 — 不需要
推荐SDD工具GitHub Spec Kit + 华为CodeArtsOpenSpec + Cursor（融合策略）OpenSpec（轻量模式）或纯Agent Coding

场景间互斥分析：场景A的"高规格+低速度"需求与场景C的"低规格+高速度"在方法论层面不可调和。任何声称"SDD适合所有场景"的论断都是错误的。Böckeler的核心发现正是：当前SDD工具没有提供适合不同问题规模的工作流灵活性。这说明SDD工具本身需要场景自适应的模式切换（如pragspec的FULL/FAST/SHORT-CIRCUIT三模式）。 需求权重量化：如果我们按全球AI编码工具市场分布（92%美国开发者使用AI编码工具日常），假设场景分布为A:15%, B:45%, C:40%（粗略估计），那么SDD的"甜蜜区"（场景A+B的部分区间）约占市场的40-50%。但场景C的40%用户如果被强制使用SDD，效率将显著下降。

1.4 步骤一交叉分析

内部张力：三个Top产品（Spec Kit、Kiro、Qoder）对"什么是SDD"有根本不同的定义。Spec Kit强调Constitution强制+分支隔离，Kiro强调IDE内嵌的会话式规格生成，Qoder强调自主代理的自学习。实际上它们在使用三种不同的"规格"概念。这暗示"SDD"作为一个市场标签可能已经语义扩散（Böckeler首次提出），标签下的产品解决的是不同的问题。

跨步矛盾：初始无知声明中"中国市场独立路径"的猜测得到初步验证——中国SDD路径确实有独立特征：金融行业先行（vs 美国的互联网/创业公司先行）、国产模型生态驱动（vs 美国的Claude/GPT驱动）、政策合规为第一推动力（vs 美国的效率/成本为第一推动力）。

论证链检验：三个独立来源（Böckeler/Martin Fowler、Scott Logic、INNOQ）都独立指出了SDD工作流"对小型任务过于笨重"的问题——这是一个被多源验证的稳健发现。 论证强度：🟢 强

1.5 核心假设（步骤一之后输出 · R5）

假设H1（效率边界假设）：如果SDD确实比Vibe Coding更能提升工程质量，那么我们应该看到：SDD团队在项目启动后第3-6个月的代码质量指标（流失率、安全漏洞密度）显著优于纯Vibe Coding团队。

假设H2（场景依赖假设）：如果SDD的价值是场景依赖的，那么我们应该看到：在需求不确定的探索性项目中，SDD组的效率低于非SDD组；在需求明确的合规性项目中，SDD组的效率高于非SDD组。

假设H3（成本幻觉假设）：如果SDD的隐形成本（规格编写+维护+审查）被低估，那么我们应该看到：在纳入规格维护成本后，SDD项目的总工时超过纯AI辅助编码项目。

2.1 范式深度拆解（R18）

范式一：Spec-Driven Development (SDD) —— 规格驱动开发

(a) 组件清单

(b) 运行步骤（以Spec Kit为例）

  1. 初始化项目（/speckit.constitution）：输入=项目愿景/约束 → 处理=LLM生成初始宪章草案 → 输出=constitution.md → 决策点：团队审议+批准 → 瓶颈：需要资深工程师参与审议
  2. 编写规格（/speckit.specify）：输入=功能描述+constitution.md → 处理=LLM生成用户故事+验收标准+边界条件 → 输出=spec.md → 决策点：团队审查通过或返回修改
  3. 澄清模糊点（/speckit.clarify）：输入=spec.md → 处理=LLM识别不明确处并生成问题 → 输出=澄清列表 → 决策点：人工回答每个澄清问题 → 瓶颈：此步骤高度依赖人工判断
  4. 生成技术方案（/speckit.plan）：输入=spec.md+constitution.md+代码库上下文 → 处理=LLM分析现有架构+生成技术设计方案 → 输出=plan.md → 决策点：技术评审
  5. 任务分解（/speckit.tasks）：输入=plan.md+spec.md → 处理=LLM依赖分析+任务拆分 → 输出=tasks.md（排序、独立可测）
  6. 一致性检查（/speckit.analyze）：输入=所有Markdown工件 → 处理=跨文档规则检查 → 输出=合规报告 → 瓶颈：当前版本为静态检查，非语义级
  7. 执行实现（/speckit.implement）：输入=tasks.md+所有前置工件 → 处理=AI代理按tasks逐步执行 → 输出=代码+测试 → 决策点：每个task完成后通过CI验证 → 瓶颈：LLM大上下文窗口中的指令遵循率不稳定
  8. 验证与修复循环：输入=CI反馈+测试结果 → 处理=LLM分析失败+自动修复 → 输出=修复后的代码 → 决策点：超过3轮修复仍未通过则人工介入

2.3 步骤二交叉分析

内部张力：SDD和约束工程两范式之间存在一个根本性张力：SDD试图通过"前置规格"来预防问题，约束工程试图通过"后置检查"来兜底问题。这两个策略在最理想情况下应该互补，但在实践中，如果前置规格不够精确，后置检查的失败率会急剧上升，导致Agent进入"修复→失败→再修复"的无限循环——这正是Bun重写案例中13,000+ unsafe调用的根源：前置规格（Rust安全规范）写不进Agent的"肌肉记忆"。

跨步矛盾：步骤一中我们对"SDD标签语义扩散"的观察在步骤二得到技术层面验证：Spec Kit的"Constitution"、Kiro的"steering memory"、Qoder的"自学规格"三者在技术实现上几乎没有任何共享机制——它们只是都标榜"SDD"。

论证链检验：Böckeler的"Verschlimmbesserung"概念（用改进的方式把事情弄得更糟）是针对SDD框架层的最有力批评。这个批评的论证链支持：Agent Guardrails作者实测Prompt规则只有60-70%可靠性 → 依赖Prompt规则的SDD框架天然有30-40%的指令漏失 → 漏失的指令可能造成比"无规格"更差的结果（Agent死板遵循部分规则而忽略其他）。 论证强度：🟢 强

3.3 路径依赖分析

必然性：SDD的兴起具有高度必然性——当46%的新代码由AI生成时，人类必须通过"意图表达"层来维持控制，SDD就是这个意图表达层的自然形式。Cursor Rules/AGENTS.md的兴起遵循同一逻辑。

历史偶然性：

  • Spec Kit选择7步刚性工作流（而非像OpenSpec的2步灵活模式）有偶然性——它由GitHub的SDD团队设计，该团队天然倾向于"平台化+完整工作流"的产品思维。
  • 中国市场SDD的金融先行是偶然的——民生银行成为了标杆不是因为银行是最佳场景，而是因为民生银行的科技团队恰好是早期探索者。如果最早探索的是一家电商公司，中国SDD可能走不同的路径。
  • Claude Code在SDD生态中的核心地位有偶然性——它源于Anthropic对agentic能力的前期投入，而非对SDD的预先规划。

步骤五：泛化分析

5.1 范式外推

如果把SDD和约束工程两个范式推向极限，会得到什么？

  • SDD极限："Spec as Source"（Tessl尝试的方向）。如果规格语言足够精确，理论上可以视为一种极高级编程语言——人写意图，机器完美翻译。这个方向的死穴是：软件需求的本质复杂性无法被任何规格语言消除（Fred Brooks 1986年已经论证了这一点）。Tessl的关闭验证了这个极限在当前时间窗口不可达。
  • 约束工程极限："全自动治理"——每个AI生成操作都经过确定性验证门，形成闭合反馈环。这个方向的死穴是：约束规则本身需要维护，而维护规则的复杂性可能最终超过直接写代码的复杂性——这是"元治理的元治理"无限回归问题。

5.2 新组合：SDD + TDD + 约束工程的三角架构

三个范式可能的最优组合不是"哪个更好选哪个"，而是各司其职：

  • SDD管"做什么"（What）：规格定义业务意图 → 轻量模式（Spec-first，不追求Spec-as-source）
  • TDD管"对不对"（Correctness）：测试定义行为契约 → 这是确定性验证的最强形式
  • 约束工程管"好不好"（Quality）：架构规则、安全策略、合规检查 → 确定性+LLM辅助双层

当前市场上最接近这个三角架构的组合方案是：OpenSpec（轻量SDD） + Claude Code Dynamic Workflows（Agent执行+TDD内置） + Harness AI/Code Guardian（约束层）。

5.3 反向推演：如果SDD根本是错的呢？

如果SDD的前提假设——"通过前置规格可以显著提高AI代码质量"——是错的，那么可能的原因是什么？

  1. 软件的本质复杂性无法前置：Fred Brooks在《没有银弹》中论证的"本质复杂性"意味着，某些设计决策只能在编码过程中被发现。如果这个论证对AI辅助编码同样成立（目前没有证据否定它），那么SDD在本质上有一个不可逾越的上限。
  2. 规格和代码间的"翻译误差"大于Vibe Coding的"直接误差"：如果AI在理解规格时的误解率 × 误解导致的错误严重度 > AI在没有规格时的幻觉率 × 幻觉导致的错误严重度，那么SDD在统计意义上是负收益。目前没有研究直接比较这两种误差率。
  3. SDD的审查成本是一个"隐蔽冰山"：Vibe Coding的可审查单位是代码（开发者熟悉的审查对象），SDD的可审查单位是Markdown文件（质量参差、太长、非可执行）。如果规格审查的单位边际成本 > 代码审查的单位边际成本，那么SDD的"前置设计节省的返工"可能被"规格审查额外开销"抵消。

5.4 沉默证据分析

  • 用户报道的SDD失败有多少未公开？企业SDD失败的内部复盘几乎不可能公开。我们看到的Saleforce和阿里Qoder案例无疑是成功案例——失败案例的不可见性造成严重的幸存者偏差。
  • GitHub Spec Kit的115K星中有多少是"挂星"而非"使用"？开源项目的star-to-active-user比率通常在10:1到100:1之间。如果按20:1估算，Spec Kit的实际活跃用户约5,750个团队。
  • 中国市场有多少SDD探索因为模型能力不足而放弃？这个问题目前没有任何公开数据，但从通义千问与Claude Sonnet在代码基准测试中的差距推测，中国市场的SDD可能有更高的失败率和更低的规格→代码转化质量。

5.5 更优解空间：将差距拆为三类约束

约束类型当前状态可突破性
物理约束LLM的本质非确定性、上下文窗口限制（虽然已扩展至百万token级但指令遵循率仍不完美）低——这是基础模型层面的限制，不是SDD框架能解决的

步骤六：收敛

6.1 假设终检（R5）

H1（效率边界假设）检验结果：部分证实，但证据不对称

证实的方向：GitClear数据（代码流失率从3.3%到7.1%）和CodeRabbit数据（AI代码1.7x问题）为"Vibe Coding损害长期质量"提供了强有力的定量支持。Salesforce案例（有效产出+151.3%+incident下降5%）提供了"规则驱动的AI生成可能同时提升产量和质量"的正面案例。

未证实的方向：没有找到SDD vs Vibe Coding的受控AB测试数据。所有现存数据都是观察性的或有选择偏差的。我们无法排除"Salesforce之所以成功是因为它是一个成熟工程组织，而非因为SDD本身有效"。

本结论最可能出错的原因是：缺乏直接的SDD vs Vibe Coding对比研究——所有证据都是间接的。

H2（场景依赖假设）检验结果：被多个独立来源反复强化

强化的证据：

  • Böckeler：SDD工具对小bug修复是"用大炮打蚊子"（独立评测）
  • Scott Logic："没有SDD我的效率大约高出10倍"（独立评测）
  • 阿里淘特实践：采用"融合策略"而非纯SDD——轻量级技术方案模板+Rules+Agent Coding（自力更生的务实妥协）
  • 民生银行策略：从"Spec-first"切入，向"Spec-anchored"演进——承认不是所有场景都适合深规格
  • INNOQ：需求和设计大量依赖隐性知识，无法前置写入规格

结论：H2被高度证实。本结论最可能出错的原因是：场景分类本身的主观性——我们对场景A/B/C的赋值（准确率5/3/2）和权重估计（15%/45%/40%）缺乏实证校准。

H3（成本幻觉假设）检验结果：未被完全证实但获得强力间接支持

支持证据：

  • Microsoft撤销Claude Code许可的根因是token成本失控（总成本=员工工资+token账单，而token账单最终超过员工成本）
  • Uber 2026年预算4个月耗尽（$3.4B），没有明确的产出提升对应
  • GitClear：9x更多代码流失（"开发者不是在更快构建，而是在不断纠正重复的错误代码"）
  • METR：19%减速——验证/修复AI输出的时间可能超过省下的编码时间
  • INNOQ实测：简单功能产生1,070行Markdown——规格审查成本可能超过直接编码

但H3无法被完全证实的原因：缺乏将"规格编写+审查+维护时间"单独核算的研究。我们只能推断隐形成本"可能很大"但不知道具体有多大。本结论最可能出错的原因是：如果组织的规格复用率很高（同一个Constitution/模板用于多个项目），H3的"总工时超过"可能不成立——但我们缺乏复用率数据。

步骤七：红队反证时刻（R6）

7.1 红队攻击 #1：SDD的"成功案例"都是幸存者偏差

攻击目标：所有声称SDD有效的成功案例

攻击内容：Salesforce（151.3%增长）、阿里（5人7天）、Bun（11天重写）、民生银行（70%测试覆盖率）——这些"SDD成功案例"全是精心挑选的成功故事。我们用前六步收集的证据来反证：

  • Salesforce是Anthropic的最大企业客户之一，发布了Claude Code的"成功故事"——这是商业合作产物，不是独立验证。且Salesforce的"有效产出+151.3%"指标是公司自研ML模型计算的，计算方法不公开。
  • 阿里的5人7天案例是"用Qoder构建Qoder自己的产品"——团队对工具的熟悉程度远超外部用户。
  • Bun重写案例的代码质量争议（13,000+ unsafe调用 vs 同类项目uv的73个）被严重淡化了——这不是"SDD成功"，这是"AI生成了量巨大的代码，但质量巨大存疑"。
  • 民生银行是公开报道中中国的唯一深度案例——如果SDD这么好，为什么没有更多银行快速跟进？

严重程度：颠覆——如果所有"成功案例"都是高度自选的幸存者，那么我们对SDD有效性的整个证据基础可能比我们想象的脆弱得多。

受影响的前序步骤：步骤一（场景调研——需要重新评估所有厂商案例的可信度）、步骤六（假设终检——H1和H2的置信度需要下调）

7.2 红队攻击 #2：SDD的Constitution机制产生"伪安全感"

攻击目标：SDD的Constitution/约束机制

攻击内容：Constitution机制（Spec Kit的核心卖点）可能在统计上比"无Constitution"更危险——因为它创造了合规的假象：

  • Böckeler发现Agents"频繁忽略checklist项目"或"过度热情地遵循规则"。这意味着Constitution既不完全执行也不被完全忽略——它在半执行状态下运行，而开发者可能因为Constitution的存在而降低审查强度（"反正有Constitution把关"）。
  • Agent Guardrails作者实测Prompt规则的可靠性仅60-70%。如果开发者因为信任Constitution而减少审查，那么30-40%的漏失率会直接转化为生产缺陷。
  • alexcpn/speckit_test的"依赖-Constitution陷阱"分析显示了恶意合规（malicious compliance）：AI不是在做最好的工程决策，而是在做最容易通过Constitution审查的决策——这可能导致系统性地选择次优方案。

严重程度：削弱——这不会推翻SDD的价值，但会显著削弱Constitution作为"安全网"的可信度：Constitution可能不是减少风险的机制，而是一个风险转移机制（把风险从"AI随意生成"变成了"AI在所有错误的地方遵守规则"）。

7.3 红队攻击 #3：SDD可能系统性地抑制创新

攻击目标：SDD对软件创新的长期影响

攻击内容：如果SDD成为主流，它可能系统性地抑制软件架构创新：

  • SDD要求"在写代码之前写好规格"。但过去30年最有影响力的软件系统（Linux、Git、React、Docker、Kubernetes）都不是从"规格"开始的——它们是迭代实验的产物。
  • 如果工程师在写代码之前必须先填Constitution+spec+plan+task文档，那么"试试看会怎样"的探索性编码文化会被系统性地压制——而这正是软件创新最强大的驱动力。
  • 更隐蔽的是，SDD的"规格先行"会强化"已知-已知"问题的解决能力，但系统性地压制"未知-未知"问题的发现——而后者是真正的技术突破的来源。

严重程度：削弱——这是一个长期风险，短期内没有直接证据。但如果SDD在10年内成为行业标准，它对软件创新的抑制作用可能会比任何短期效率提升要大得多。

7.4 红队攻击 #4（框架层）：这个分析框架本身就预设了"SDD是值得研究的"

攻击目标：整个调研的分析框架

攻击内容：这个调研框架的两个盲区：

  1. "工具中心主义"盲区：我们的整个分析都围绕着"什么工具、什么方法、什么数据"——但真正决定AI辅助软件工程质量的，可能不是工具或方法，而是团队的社会学因素（信任水平、心理安全感、工程师是否敢于质疑AI输出）。一个没有心理安全感的团队，即使有最好的SDD工具，工程师也不会主动指出"这个规格有问题"——而SDD的全部价值恰好依赖于这种主动质疑。我们完全忽略了组织行为学维度。
  2. "工程vs经济"错位：我们把"token成本不可持续"作为一个工程问题来分析，但实际上它是一个商业模式问题——AI厂商的定价策略（订阅vs用量、基础模型vs应用层利润分配）才是根本原因。如果Anthropic明天把Claude Code的价格降为现在的十分之一（技术上可能，商业上不愿），那我们的"成本幻觉假设H3"可能瞬间被推翻——而我们的分析框架完全没有考虑这个可能性。

严重程度：颠覆——这两个盲区意味着我们可能在分析一个本质上不是"技术问题"的问题时，使用了纯技术框架。如果决定SDD实际成功的关键因素是组织文化+定价模式而不是技术设计，那么我们70%的分析都是打在错误的目标上。

步骤八：结论与行动建议

8.1 结论分层输出

🔴 立即行动

行动1：如果你在用Vibe Coding，立即引入轻量SDD + 确定性CI门禁

这不是建议，这是判断。

判断：Vibe Coding的长期技术债务已经被多个独立来源量化证实（GitClear代码流失率+115%、CodeRabbit 1.7x问题、DZone 45%安全漏洞率）。纯Vibe Coding在3个月后产生的维护成本 > 轻量SDD的规格编写成本。这不是"应该做"，而是"不做会付更高的代价"。

什么场景下选：任何不是"一周内丢弃的原型"的项目。如果你预期代码会存在超过3个月，就应该引入至少最小化的规格实践（AGENTS.md + OpenSpec轻量模式 + lint/test CI）。

天花板在哪：轻量SDD不能解决架构级问题（需要约束工程层的ArchUnit/Code Guardian补充），不能解决LLM本身的能力边界（物理约束）。

什么时候该换：当你的团队规模超过10人，或者你需要满足合规/审计要求时，从轻量SDD升级到完整SDD（Spec Kit或CodeArts）。

行动2：如果你的团队在使用AI Agent，立即建立token成本监控和预算上限

判断：Microsoft（100K工程师撤销许可）、Uber（$3.4B预算4个月耗尽）、NVIDIA（"计算成本远超员工成本"）三个独立案例共同证明：不加监控的Agent使用会导致token成本失控。这不是"可能发生"，而是"已经在三个标志性公司发生了"。

什么场景下选：任何使用Agent模式（非简单代码补全）的团队，无论大小。

天花板在哪：成本监控不能解决Agent效率问题（~70% token被浪费在重复读取和死胡同探索），但至少能让你知道问题的规模。

🟡 趋势观察

趋势1：SDD将分化为两条路径——"企业合规SDD"和"轻量灵活SDD"

判断：市场正在分裂。Spec Kit/Qoder/CodeArts正在走向"企业合规SDD"（完整工作流+审计+Constitution），OpenSpec/Cursor+自定义Rules正在走向"轻量灵活SDD"（最小规格+多模式+工具灵活组合）。Tessl的关闭证明了第三条路（Spec-as-source的极端规格）在当前时间窗口不可行。两条路径将共存，服务于不同的场景。

建议：不要等待SDD市场"成熟"——两条路径的分化已经是事实。现在就根据你的场景选择一条路径，而不是等待某个"终极统一方案"出现。

趋势2：AI编码工具的定价模式将在2026-2027年经历结构性变革

判断：GitHub Copilot从固定订阅转向用量计费（2026-06）、Microsoft从Claude Code撤退、Uber预算爆炸——这三个信号同时表明：当前的AI编码定价模型是不可持续的。未来12个月的变革方向可能是：(a) 更多"AI-native"定价（按PR/按feature/按有效的代码变更而非token消耗），(b) 更严格的用量上限和预算控制工具，(c) 开源/自托管方案的崛起。

🟢 风险警示

风险1：Constitution/约束机制产生"安全假象"——降低审查质量

判断：我们观察到一种反直觉的风险模式：SDD的Constitution机制可能增加缺陷率，因为它降低了人工审查的警惕性。当开发者认为"反正有Constitution把关"时，他们可能倾向于批准未仔细审查的PR。当前的实证数据（Agent Guardrails：Prompt规则60-70%可靠性）不支持将Constitution作为主要防线。

缓解措施：将Constitution定位为"辅助工具"而非"安全网"。审查者在批准PR前必须阅读代码，而不仅仅是检查"Constitution合规清单"。在CI门禁层面引入双通道验证（确定性规则+人工抽样代码审查）。

风险2：SDD可能放大而非缩小组织中的"隐性知识鸿沟"

判断：SDD假设需求可以被显式化为规格文档。但在现实中，组织中最有价值的知识往往是隐性的（tacit knowledge）——资深工程师的直觉、对系统脆弱点的感知、对"什么看起来不对"的模糊判断。如果SDD让组织过度依赖显式规格，那些无法被显式化的判断会被系统性地排除在开发流程之外——而红队攻击#4指出了这个框架层盲区。

8.2 回头对无知声明

初始不确定项现在有答案了吗？
SDD的"隐性成本"规模有多大？ 部分有答案。我们确认了隐形成本的存在（规格编写+审查+维护），通过间接数据（Microsoft/Uber成本危机、INNOQ的1,070行Markdown案例）量化了其"可能很大"。但我们仍缺乏直接的AB对比数据来量化"规格维护工时占总工时的百分比"。没有直接答案。
SDD是否在大型组织中可规模化？ 混合答案。Salesforce案例（1,000+工程师标准化Claude Code）提供了正面证据。但Microsoft的撤退（100,000工程师被撤销Claude Code许可）提供了反面证据——规模化的瓶颈不是技术而是经济。Got an answer but it's about economics, not technology.
中国市场的SDD路径是否独立？ 是，基本证实。中国市场的SDD有清晰的独立特征：金融行业先行、国产模型驱动、Qoder/CodeArts/CodeBuddy构成的封闭生态。但与全球市场的底层方法论（Spec→Plan→Tasks→Implement）是共通的。
