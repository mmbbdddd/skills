步骤一：产品/场景调研

1.1 什么是 Spec-Driven Development (SDD)？

SDD是一种将规格说明（Specification）提升为软件开发首要工件的工程方法论。其核心主张："先定规范，再写代码。规范声明系统做什么、在约束下如何行为、关键决策背后的原因——代码是规范的输出，不是真相的源头。"（Microsoft for Developers, 2026年6月）

与Vibe Coding的核心区别：Vibe Coding的人机交互是"提示-生成-修补"的碎片化循环，需求存在于聊天记录中，不可追溯。SDD将需求结构化为持久化的规范文件，使其成为AI编码代理每次决策的唯一真相源。

截至2026年6月，SDD生态已从少数几个工具发展到30+框架和工具（SoftwareSeni, 2026年5月），并获得了成熟度模型、学术论文和企业实践的支撑。

1.2 产品深度分析：Top 3

1.2.1 GitHub Spec Kit（115k+ Stars）

产品形态：开源CLI工具包（MIT许可），搭配VS Code/JetBrains扩展。面向团队开发者，核心交互为斜杠命令驱动的七步工作流。维护方为GitHub/Microsoft。

深入分析（三要素）：

  • (a) 独立评价：ThoughtWorks Technology Radar Vol.34 将SDD整体放入"Assess"（评估）环而非"Adopt"（采纳），理由是工具成熟度和团队适应成本仍需要验证（ThoughtWorks, 2026年4月）。DevOps.com指出Spec Kit是"将规范放回软件开发"的标志性尝试（DevOps.com）。
  • (b) 用户真实体验：Xebia的实战评测指出Constitution（宪法）机制是"最被低估的特性"——它将团队约定从口头传统变为AI可读取的铁律（Xebia Blog）。SDD框架对比评测（RanTheBuilder, 2026年2月）给Spec Kit综合评分2.77/5，低于OpenSpec（4.00/5）和BMAD Quick（3.74/5），主要失分在"过度工程化"和"学习曲线"。论证强度：中（🟡）——评测者为个人博主，方法论未披露
  • (c) 已知局限：社区反馈的最大痛点是"spec和代码之间的漂移（drift）"——AI实现在执行过程中偏离spec后，没有自动化机制将其抓回。Spec Kit的答案（/analyze 和 regenerate-from-spec）被认为"太过粗暴"（Godel Technologies, 2026年5月）。此外，面向已有系统的增量修改（brownfield场景）体验远不如OpenSpec（Will Torber, 2026）。

地理市场：全球，英语为主。中文社区衍生品SpecKit-CN已发布（GitHub）。

1.2.2 AWS Kiro

产品形态：付费AI原生IDE（独立或VS Code兼容），信用点数计费（$20-$200/月），底层为Amazon Bedrock。面向AWS生态企业开发者。核心功能为需求规范→设计→任务分解→自动实现→验证。

深入分析（三要素）：

  • (a) 独立评价：GeekWire指出Kiro的Requirements Analysis功能是"对AI代理可靠性质疑的直接回应"——AWS在经历Financial Times关联报道后加速了该功能（GeekWire, 2026年5月）。InfoWorld评价其Bugfix Spec功能为"SDD在已有项目中落地的务实路径"（InfoWorld）。
  • (b) 用户真实体验：Devoteam评测确认Kiro的Spec Mode在复杂特性开发中"Vibe Mode的代码质量明显更高"，但也指出"在Vibe Mode和Spec Mode之间切换有时意味上下文丢失"（Devoteam）。
  • (c) 已知局限：闭源生态是关键局限——不可审计的SMT求解器、代码和规范分散在多个文件夹中导致功能跟踪困难（vs OpenSpec的集中式变更管理）、AWS锁定风险。中文媒体报道指出"Kiro更适用于AWS原生场景，非AWS生态迁移成本高"（财经移动）。

地理市场：全球，AWS GovCloud已支持（2026年2月）。中国市场通过AWS中国区可用。

1.2.3 Qoder（阿里云）

产品形态：阿里云推出的Agentic AI编码平台，支持IDE、CLI、VS Code/JetBrains插件。底层为阿里云百炼大模型。核心功能为三种Quest模式（Spec驱动开发/搭建网站/原型探索），包含独创的"5人7天"产品化流程示范。

深入分析（三要素）：

  • (a) 独立评价：阿里云官方强调Qoder是"国内首家将Spec-Driven作为核心范式的AI编程平台"（Qoder官方Blog）。雷锋网报道Quest 1.0的自进化智能体技术为"能在深入分析代码后实现自我学习"（雷锋网, 2026年1月）。
  • (b) 用户真实体验："5人7天"案例提供了一条端到端的SDD实践轨迹——Day 0只写Spec（零行代码），第1-2天AI开发，第7天上线。该案例的局限在于它是工具开发者自己做的示范，存在明显的"自我验证"偏见——但流程本身的详细记录（从what到how的每一步）仍提供了有价值的参考。
  • (c) 已知局限：作为阿里云生态的组成部分，Qoder的非阿里云迁移成本较高。社区反馈和第三方评测远少于Spec Kit和OpenSpec。Quest模式的适用场景目前限于Web应用和标准后端，复杂系统的支持尚未见公开案例。

地理市场：中国为主，通过阿里云国际站向海外延伸。R12：中国地理覆盖已满足

场景间互斥分析场景A和场景C在准确率和速度维度上互斥——追求A的工具引入大量前摇（宪法定义、规范澄清、评审周期），这对C场景是致命的。反之亦然：C的速度在A场景中会产生不可接受的技术债务。这解释了为什么"一个工具统治所有"在SDD中是不可取的——R9决策矩阵将体现这一点。

Gate 1a 检查——意外发现（≥3）：

  1. 意外发现1：AI让有经验的开发者变慢19%（METR），而非变快。这与几乎所有厂商宣传相反。METR的试验设计（严格RCT，16位有经验开发者，$150/小时，246个真实任务）对AI编码的生产力提升叙事构成了最有力的经验挑战。
  2. 意外发现2：CMU研究发现AI采纳带来的速度提升在3个月内完全消退。+281%的代码行增量在第二个月就降至+48%，第三个月回到基线——但代码复杂度和警告数的恶化持续存在。这意味着AI编码工具有可能是一场"速度幻觉"——短期快速增长被长期技术债务完全抵销。
  3. 意外发现3：Kiro的SMT求解器发现60%的第一稿需求包含逻辑错误。这个数字对"Spec驱动开发"自身的基础假设构成了讽刺性的挑战——如果需求规范本身就是不可靠的，那么"规范驱动"从源头就可能错了。SMT求解器的必要性反过来证明了人类写需求也是不可靠的。
  4. 额外发现：SDD生态内部的"同一性错觉"。表面上所有工具都声称在做SDD，但它们对"Spec"的定义截然不同——Spec Kit定义为分阶段的宪法+功能规范+计划+任务，OpenSpec定义为增量变更提案，Kiro定义为需求+设计+任务三文档。没有统一的Spec交换格式，意味着从一个SDD工具迁移到另一个几乎等于重写。

Gate 1b 检查——场景分解完成（R7）：✅ 3个场景已分解，需求权重已量化（1-5分），竞品替代方案已列出，互斥分析已完成。

1.4 交叉分析：内部张力/跨步矛盾/论证链检验

内部张力：SDD工具声称自己解决了Vibe Coding的问题，但每个工具本身都面临spec漂移的同一困境。冲突在于：一个声称要消除不确定性的方法论，自身最大的问题就是不确定性（spec和代码的偏离）。论证强度：中（🟡）

论证链检验：Vibe Coding代价数据（GitClear + CMU + DORA）构成了SDD必要性的证据链——但这个链条有一个隐藏假设：SDD能有效减少技术债务。这个假设目前只有一篇论文（Constitutional SDD, 73%减少）的单一数据点支持，且该论文作者与研究利益无关性的验证不足。论证强度：中（🟡）——等待多源验证

1.5 核心假设（R5）

假设1（增强假设）：如果SDD显著减少了AI生成代码的技术债务，那么我们应该看到采用SDD的团队在6个月后的代码重构率（moved code %）显著高于纯Vibe Coding团队。

假设2（限制假设）：如果SDD的生产力成本（规范编写+审查时间）超过AI编码节省的时间，那么SDD在非合规场景下的净生产力增益应该是负的或微弱的。

假设3（悖论假设）：如果SDD的效果取决于规范的质量，而人类写规范也有高错误率（60%），那么SDD的实际效果上限由规范质量的上限决定——SDD并没有解决"人类无法精确表达需求"这一根本问题。

步骤二：技术范式/原理/实现

2.1 范式识别

基于前序调研，AI辅助软件工程的"设计优先"实践分为两种本质不同的技术范式：

  • 范式I —— 管道式规范-生成（Pipeline SDD）：规范撰写 → AI根据规范生成代码 → 人工或自动验证。代表：GitHub Spec Kit、BMAD、OpenSpec、Qoder Quest模式。
  • 范式II —— 形式化约束-验证（Formal-Verified SDD）：规范撰写的同一阶段引入形式化验证（SMT求解、架构规则自动检查），在代码生成前消除规范中的逻辑矛盾。代表：AWS Kiro Requirements Analysis、Constitutional SDD（arXiv 2602.02584）、Archgate。

(b) 运行步骤/流程

  1. Constitution定义：输入=团队约定，处理=模板化编码为AI可读规则，输出=constitution.md。瓶颈：需要团队共识，1-3天
  2. 需求收集：输入=产品需求描述，处理=与stakeholder澄清，输出=非结构化需求文档。
  3. Spec生成：AI根据constitution+需求生成结构化spec，包含用户故事、验收场景、边界条件。输出=spec.md。瓶颈：质量依赖原始需求准确度，60%可能含逻辑错误
  4. Clarify（可选）：AI检测spec中的模糊点，提出二选一问题，人工解答。
  5. Plan生成：AI设计架构，选择技术方案，绘制数据流图。输出=plan.md。决策点：是否需要对plan进行人工评审
  6. Task分解：plan分解为任务列表，建立依赖关系。输出=tasks.md。瓶颈：大项目任务可达数百个，依赖关系可能出错
  7. 逐任务实现：AI按依赖顺序执行每个task，每个task产生代码+测试。
  8. 跨文件一致性分析：自动化检查spec-plan-tasks-code的一致性。
  9. 人工审查+合并：最终质量闸门。

(b) 运行步骤/流程

  1. 需求撰写：人类（或AI辅助）撰写初始需求。输入=业务需求，输出=自然语言需求文档。
  2. LLM精准化：LLM将模糊的自然语言转为EARS（Easy Approach to Requirements Syntax）格式的精确需求。瓶颈：LLM可能在精准化过程中引入误解
  3. 形式化转译：EARS格式转为SMT求解器可处理的一阶逻辑公式。
  4. SMT求解：数学证明引擎检查：矛盾（两个需求在同一条件下要求不同行为）、歧义（可被不同理解）、缺口（未覆盖的边界条件）。关键决策点：SMT求解是NP-hard问题，大需求集可能超时
  5. 问题生成：求解结果转为自然语言的二选一问题供人类快速回答（~15秒/题）。
  6. 人工修正：人类修订需求后重新运行SMT求解。
  7. 规约冻结：矛盾解决后，生成最终spec作为AI实现的不可变契约。
  8. 代码生成+验证：AI按spec生成代码，CI/CD管线中的架构规则引擎同步检查。

  • SDD工具的分裂（30+框架）而非统一——如果有巨头推出一体化标准，这个数字会迅速收敛

步骤五：泛化分析

5.1 范式外推

从SDD到"意图工程"（Intention Engineering）的外推：

如果当前"Spec规范+AI生成代码"的管道模式成立，那么外推的终点是什么？一组可能的演化路径：

  • Spec-as-Source（Level 5成熟度）：人只编辑Spec，代码完全从Spec生成且从不手工修改。这需要Spec编译器（如archiet-microcodegen）。当前状态：技术存在但生态不成熟——无人有大规模生产使用经验。
  • Spec Marketplaces：可复用的Spec模板（"用户认证+2FA+OAuth"作为一个规范包）像npm包一样被引用。Tessl已经在这一方向布局（10,000+ spec注册表）。
  • 反向生成Spec：AI从已有代码库逆向生成Spec文档，作为"理解已有系统"的第一步。已有工具（如代码到架构图的转化）但作为SDD的标准流程尚未成熟。

5.2 新组合

SDD + Mutation Testing（变异测试）的未探索组合：

如果有AI生成的代码通过了所有测试——但测试本身也是AI生成的（这是当前SDD流程中的常见做法）——那么代码和测试之间可能存在共谋：AI生成的测试只验证AI生成的代码的"快乐路径"。变异测试（对代码进行小修改，验证测试是否能捕获）可以作为打破共谋的额外安全网。当前没有任何SDD工具内置变异测试。

SDD + 数字孪生的潜在组合：

需求规范中的验收标准如果能在数字孪生环境中进行"压力测试"（模拟高负载、异常输入等），可以在代码生成前验证需求本身的鲁棒性。这比SMT求解器更进一步——SMT解决逻辑矛盾，数字孪生解决"在真实环境中会发生什么"。

5.3 反向推演

如果SDD的核心假设是错的：

SDD假设人类能写出更好的规范。但数据（Kiro: 60%需求含bug）暗示这个假设可能不成立。如果是这样，那么正确的解决方案不是"先写好规范再生成代码"，而是：

  • 方案A：双向同步——代码在实现过程中发现的问题能自动回流到规范中（living specs）
  • 方案B：规范层面的持续形式化验证——将SMT求解器从一次性检查变为持续集成中的常规步骤
  • 方案C：放弃规范作为核心工件——回到TDD的极端版本：只写测试，AI生成代码以满足测试，测试本身就是可执行的规范

如果Vibe Coding的成本被高估：

GitClear的方法论存在一个已知局限：它分析的是代码变更模式，而非开发者行为意图。复制粘贴的增加可能是AI工具本身的交互模式（Chat→复制到IDE），而非代码质量下降。如果CMU的发现（3个月后速度回归基线但复杂度持续增加）是真实且普遍的，那么Vibe Coding的成本就不是高估，而是低估——因为长期成本尚未完全显现。

步骤七：红队反证时刻（R6）

红队攻击 A：整个SDD叙事建立在不可靠的数据基础上

攻击目标：SDD的合理性论证

攻击内容：支撑SDD必要性的每一个关键数据点都有严重局限：(1) GitClear的数据来自2020-2024年——在SDD工具（2025年10月后才出现）出现之前，完全不能反映SDD的效果。(2) CMU研究只覆盖Cursor的.cursorrules采用者（一个非常弱的"SDD前身"），不能泛化到全栈SDD工具。(3) DORA的7.2%稳定性下降是所有AI采纳的影响——其中大部分是Copilot/Cursor用户，而非SDD用户。(4) METR的19%减速实验用的是Cursor Pro + Claude 3.5/3.7，而非SDD工作流，而且56%参与者从未用过Cursor。结论："Vibe Coding有害"的证据被系统性地用于论证"SDD有益"，但这两个命题在逻辑上独立——A坏不意味着B好。

红队攻击 B：SDD是新一轮"形式化方法论泡沫"

攻击目标：SDD作为范式转型的持久性

攻击内容：软件工程史上有过多次"如果你先把设计做好，实现就会自动正确"的尝试——从UML MDA（模型驱动架构，2000年代）到形式化方法（1990年代至今）到Design-by-Contract（Eiffel, 1980年代）。每一次的瓶颈都是同一个：形式化的成本在前端（需求/设计），收益在后端（实现/维护），而软件项目的前端永远比后端更仓促、预算更少。SDD如果不是重复同一个模式，它必须解决"前端成本"问题——但BMAD Full 6天/功能、Scott Logic "spec编写数小时生成数分钟"的数据表明它没有。如果历史有任何预测力，SDD将在3-5年内回归为一个"在特定场景下有用"的方法论，而非行业主流。

严重程度：削弱——不使SDD的局部价值无效，但质疑其作为"行业主流"的可持续性。

攻击内容：本报告的分析框架有三重盲区：

  1. "工具/方法论"偏误：将SDD视为工具/方法论选择问题，忽略了组织层面——SDD是否有效的最大决定因素可能不是工具选择，而是团队是否拥有能写清楚Spec的产品经理。这超出了本报告的分析范围。
  2. "西方中心"盲区：中国市场的数据主要来自厂商（阿里云、华为）和受监管行业（银行）。中国中小团队和互联网公司的SDD采用数据完全缺失。这可能不是因为不存在，而是因为本报告搜索方法论中的语言和渠道偏误。
  3. "现在"盲区：本报告的数据截止2026年6月。3-6个月后可能会有决定性变化——Kiro SMT公开发布后的独立测试、更多SDD vs Vibe Coding的头对头研究、EU AI Act生效后的合规需求爆发。本报告的结论可能在很短时间内过期。

严重程度：颠覆（对框架）——不是对SDD的颠覆，而是对本报告分析完整性的颠覆。这意味着报告的"不知道什么"声明（前置一）需要被郑重对待。

红队攻击 D：SDD解决的是"错误的问题"

攻击目标：SDD的"问题陈述"

攻击内容：SDD声称要解决"AI生成代码质量差"的问题。但如果高质量代码是目标，为什么不直接投入AI的训练数据质量改进、模型架构优化、和更好的代码审查工具？SDD本质上是在使用者层面而非工具层面解决质量问题——这隐含地将责任从工具提供者转移到工具使用者。这有可能是在为不完善的AI工具开脱："你的AI生成代码有质量问题不是AI的问题，是你没有写好Spec。"

严重程度：削弱——不影响SDD功效的证据，但重新框架了"谁为质量问题负责"的归因。

8.1 结论分层输出

🔴 立即行动（1-2条）

行动1：在合规敏感或安全关键的新项目中试点Paradigm II（形式化验证SDD），优先选择Kiro（如果已在AWS）或Constitutional SDD框架（如果需要开源/可审计）。

这是判断不是描述：选它，但要限域。理由：EU AI Act高风险管理义务2026年8月2日生效，罚款最高3500万欧元或全球营业额7%。Constitutional SDD的73%安全缺陷下降（虽然论证强度仅为🟠弱）是目前唯一有量化数据的合规路径。优先场景：金融系统核心组件、医疗数据处理、身份认证。不在以下场景使用：原型探索（Vibe Coding更快）、非关键UI（SDD仪式感不值得）。

天花板：SDD本身不能保证合规——它提供的是一条可追溯的规范→代码链路，减少了可检查的漏洞，但不能消除未知的漏洞。

什么时候该换：当市场出现被独立审计机构（如BSI或TUV）认证的AI代码合规工具时，SDD在该场景下可能降格为辅助。

本结论最可能出错的原因是：Constitutional SDD的73%数据未被独立复现。如果未来独立研究发现实际效果远低于此，这一行动建议的投资回报率可能为负。

🟡 趋势观察（1-2条）

趋势1：SDD工具将在18个月内经历一场"统一标准vs锁定差异化"的正面冲突。

目前30+框架的状态不可持续。两个可能的结局：(a) GitHub Spec Kit成为事实标准（因为它兼容30+Agent，生态最强）；(b) 出现一个统一的Spec交换格式（类似OpenAPI之于API规范），允许跨工具互操作。趋势观察者应关注：是否出现类似OpenAPI initiative的跨厂商Spec标准工作组。如果没有，两年后的SDD生态将锁定当前混乱。

趋势2：Vibe Coding和SDD将不会消失哪一个——而是分层共存。

Karpathy自己的演化（从"vibe coding"的命名者到"agentic engineering"的倡导者）表明了这一点：探索阶段用vibe，生产阶段用engineering。两者不是替代关系，是分层关系。这意味着投资"Vibe Coding→SDD的无摩擦切换"可能比投资"更好的SDD"更有价值。

本结论最可能出错的原因是：如果AI编码模型的质量在12-18个月内实现突破性改进（如Veracode安全通过率从55%跳升至85%+），那么"Vibe Coding有害"的数据基础可能会被推翻，分层共存的需求也会减弱。

🟢 风险警示（1-2条）

风险1：在AI编码普及但SDD缺失的环境中，技术债务以"可视化延后"的方式积累。

CMU数据显示AI采纳后3个月速度回归基线但复杂度和警告持续升高——这意味着债务积累模式与速度指标的脱钩。如果你的团队只监控速度（PR数、部署频率），你会在债务雪崩之前看不到任何警告信号。建议：立即将静态分析警告数、代码复杂度和重构率纳入工程健康仪表盘，与DORA指标并列。

风险2：中国市场SDD生态的"信息孤岛"风险。

Qoder和华为CodeArts的SDD实现与国际开源SDD工具（Spec Kit, OpenSpec）不兼容。如果中国监管机构推出自己的AI代码合规框架并与国产SDD工具绑定，中国开发者的实践将与国际社区脱钩——重演移动互联网时代Android vs iOS/中国Android分叉的历史。影响范围：需要全球合规的企业（如同时面对EU AI Act和中国监管）。

本结论最可能出错的原因是：中国企业的SDD实践可能比开源社区文档描述的更加国际化——公开文档可能不反映实际操作。
