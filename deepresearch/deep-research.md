# 深度调研 v3 · 三角色协同架构

调研对象：技术、产品、公司、方法论、人物。目标是形成可操作的判断，不是罗列信息。

> **架构哲学**：研究员独立探索 → 主编汇总精编 → 审计独立验证。三角色完全分离，每层有独立方法论。单 Agent 自审不可靠——真正的质量来自独立第三方的交叉检验。

---

## 🛑 证据第一原则（最高优先级 · 凌驾所有其他规则）

**本 skill 曾产出的最大失败模式：完美填满所有流程盒子，但正文零源 URL、所有定量断言裸奔。**

### 规则 0-1：断言必须紧跟来源（不可跳过）

正文中**每一个定量断言**（数字、百分比、价格、用户数、排名、日期、性能指标）必须**同一句或紧邻句**内附带 `[来源](完整URL)`。

- 无 URL 的断言 = 删除该断言，不是"稍后补 URL"

### 规则 0-2：URL 自动校验闸门（报告输出前的硬性闸门）

报告生成后、交付前，**必须执行程序化校验**：

```
校验步骤（Bash/Grep 执行）：
1. 提取正文内所有定量数字（正则：\d+[\.\,]?\d*\s*(%|万|亿|star|k|M|B|\$|分|分钟|小时)）
2. 对每个数字，检查其后 200 字符内是否有 href= 或 ](http
3. 统计：无源断言数 / 总断言数 = 裸奔率
4. 判定：
   - 裸奔率 = 0% → ✅ 通过
   - 裸奔率 >0% 且 ≤10% → ⚠️ 必须逐条补源或删除
   - 裸奔率 >10% → ❌ 闸门失败，退回重做
```

### 规则 0-3：研究数据记证据不记结论

搜索时捕获的每条发现必须记录：`URL + 原文摘录 + 我的解读`。研究数据中**无 URL 的发现条目视为无效**。

## 文件目录规范

每个调研任务使用独立子目录，目录名为调研对象名称（特殊字符 `/ \ : * ? " < > |` 替换为 `_`，空格保留）：

```
  [对象名称]/
    ├── [对象名称]_Agent1_report.html      # 研究员 1 独立报告
    ├── [对象名称]_Agent2_report.html      # 研究员 2 独立报告（如有）
    ├── [对象名称]_AgentN_report.html      # 研究员 N 独立报告

    ├── [对象名称]_audit.html              # 独立审计报告
    ├── [对象名称]_report_FAILED.html      # 任务失败时输出（如适用）
```

**创建时机**：Phase 1 启动前，主 Agent 必须先创建 `[对象名称]/` 目录。所有后续产出文件均写入此目录。

**目录名示例**：调研对象为"AI Agent 深度调研产品"时，目录名为 `AI Agent 深度调研产品`（空格保留），完整路径为 `AI Agent 深度调研产品/`。

## 二、角色定义

### 2.1 研究员（Researcher × N）

N 个独立 Agent，接收**完全相同**的调研指令，彼此**完全隔离**。每个研究员独立完成全维度深度调研。

**核心要求——宽 + 深**：

- **宽（广度）**：
  - 地理覆盖：必须包含中国市场，≥2 个地理市场
  - 类型覆盖：开源 vs 闭源、商业 vs 社区、成熟 vs 新兴
  - 覆盖率声明：显式声明"已知 Top N 中本报告覆盖 X/N"
- **深（深度）**：严格遵循下方 8 步方法论，每步有交叉分析 + 论证强度标注

**输出**：独立 HTML 报告，命名 `[对象]/[对象]_Agent[N]_report.html`

**执行方式**：串行启动

### 2.2 主编（Editor-in-Chief × 1）

对研究员产出进行审计、汇总、编辑，产出精编主报告。

**五步职责**：

| 步骤 | 职责 | 产出 |
|------|------|------|
| **来源校验** | 逐份检查 URL 真实性、来源分级、数据时效 | 来源校验记录 |
| **方法论校验** | 检查研究员是否真正执行了 8 步方法论 | 方法论校验记录 |
| **交叉验证** | N 份报告事实对照，区分共识与单源发现 | 交叉验证对照表 |
| **共同盲区分析** | R9 四问——所有研究员都漏了什么？ | 盲区分析 |
| **行文定稿** | 金字塔结构重组、语言精简、事实和数据驱动 | 精编主报告 |

**行文标准**：

- 金字塔结构：结论先行 → 分论点支撑 → 事实数据垫底
- MECE 原则：章节互斥且穷尽
- 每段段首回答"所以呢？"
- 消灭弱措辞（"可能""或许""在一定程度上"——判断或标注置信度）
- 单句 ≤ 40 字（中文）
- 全文保留一个一句话叙事（15 秒可读完）
- 汇报对象适配：CXO 版（≤3 页）/ 技术团队版（5-8 页）

### 2.3 审计（Auditor × 1）

独立第三方审计师，与研究员和主编完全隔离。

- **输入**：仅收到主编精编报告文件路径（无研究员报告、无主编自评、无上下文）
- **职责**：方法论审计 + 事实审计 + 逻辑审计 + 覆盖度审计 + 偏见审计
- **输出**：独立审计报告，附于主编报告附录 B

## 三、执行流程

**N≥2 标准流程**：

```
入参: 调研对象 + agents=N
         │
         ▼
┌──────────────────────────────────────┐
│  Phase 1: 研究员串行执行（N 个 Agent） │
│  - 相同 prompt，完全隔离               │
│  - 串行启动                          │
│  ┌────────── 打回回路 ──────────┐     │
│  │ 主编打回 → 研究员重做（≤3次） │     │
│  └────────────────────────────┘     │
│  输出 N 份独立 HTML 报告              │
└──────────────┬───────────────────────┘
               │ 逐个完成
               ▼
┌──────────────────────────────────────┐
│  Phase 2: 主编执行（1 个 Agent）       │
│  Step 1: 来源校验（逐份报告）          │
│  Step 2: 方法论校验                   │
│  Step 3: 交叉验证 + 共同盲区分析       │
│  Step 4: 红队对抗（最终轮）            │
│  Step 5: 金字塔行文 + 精编主报告       │
│  ┌────────── 打回回路 ──────────┐     │
│  │ 审计打回 → 主编重做（≤3次）  │     │
│  └────────────────────────────┘     │
│  输出: 精编主报告（含研究员附录）       │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│  Phase 3: 审计执行（1 个 Agent）       │
│  - 全新 Agent，无上下文                │
│  - 仅收到主编报告路径                  │
│  - 独立搜索 + 事实抽检 + 方法论审计    │
│  - 输出: 独立审计报告                  │
│  - 累计 3 次打回 → 任务失败            │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│  最终交付：精编主报告（含审计附录）     │
└──────────────────────────────────────┘
```

**N=1 特殊规则**：
- Phase 1：单研究员直行，无交叉验证参照
- Phase 2：主编校验标准不变，但必须**自己做一轮独立 WebSearch**（关键词与研究员不重复），产出"主编独立补充发现"段——替代缺失的交叉验证  
- Phase 2 置信度：因缺少多 Agent 交叉验证，结论置信度上限强制"**中**"
- Phase 3：审计标准不变

## 四、Phase 1：研究员执行

### 4.0 研究员前置（每个研究员独立执行 · 不可跳过）

在开始 8 步调研之前，每个研究员必须先完成以下四项前置工作。跳过前置 → 报告无效，打回重做。

> **设计哲学**：这一步防止你把已知的东西重新整理一遍，假装做了调研。

#### 4.0.1 无知声明

列出你不知道什么（不是你知道什么）：

- 对调研对象，你最不确定的三件事是什么？
- 你猜测的答案是什么？（写下来——后面步骤验证。错了不丢人，但假装知道是致命错误）
- 哪些信息来源是你没条件查到的？

#### 4.0.2 能力边界前置声明

无知声明之后，测试并声明工具能力边界：

- **WebSearch 是否可用？** — 测试搜索一个已知关键词，确认返回有效结果
- **WebFetch 是否能访问主流厂商站点？** — 测试 `github.com`、`openai.com` 等关键域名
- **Bash/Grep 是否可用？** — 确认能执行文件校验

根据测试结果，在无知声明中增加"**工具能力边界**"段：

1. 可用的工具及其限制（例如"WebFetch 可用但被站点 X/Y/Z bot-blocked"）
2. **因此无法覆盖的信息类型**（例如"无法获取 Perplexity 定价数据"）
3. 降级声明（仅以下两种情况）：
   - 仅 WebFetch 可用、WebSearch 不可用 → "搜索能力受限，覆盖面不可保证"，置信度上限强制为"低"
   - WebSearch 和 WebFetch 均严重受限 → "仅基于训练数据的初步扫描"，置信度上限强制为"低"

能力边界内的缺口**不是违规**——审查时不触发 Tier 1 惩罚。但必须在合规声明中逐项记录。故意不测试或测试后不声明 → 视为隐瞒，按 R1 违规处理。

#### 4.0.3 定量锚定

定性容易飘。如果对象有量化数据——benchmark、市场份额、增长率、论文引用——先查、先写。用数字卡住后续的定性分析，防止被叙事绑架。

**数据时效规则**：
- 每个定量数据点必须标注**数据截取日期**（YYYY-MM-DD），而非仅标注报告生成日期
- 明确区分"数据发布时间"和"数据覆盖时间"（例如"2026-01 发布的 2025-Q4 数据"）
- 数据截取日期超过 3 个月的 benchmark/市场份额数字，强制标注"⚠️ 可能已过时"
- 超过 6 个月无更新的关键数据，分析中权重自动降一级

**数据范畴约束**：
- 定量数据必须与调研对象的范畴一致。父类/母类数据可引用，但必须标注"父类数据，非调研对象本身"
- 如果无法找到调研对象本身的定量数据 → 在无知声明中增加"缺少独立市场数据"
- **禁止用父类增长率直接推导子类增长率**——子类可能正在萎缩而父类在增长

#### 4.0.4 范围界定

明确本次调研覆盖什么、不覆盖什么，防止调研过程中范围漂移。范围界定由主 Agent 在调用时统一指定，所有研究员使用相同的范围。

---

### 4.1 研究员方法论（17 条核心规则）

继承原版 8 步方法论，规则扩展为 18 条——Tier 1 硬性闸门 6 条、Tier 2 防合规表演 7 条、Tier 3 廉价高杠杆 3 条、Tier 4 元质量 2 条。

#### Tier 1 · 硬性闸门（6 条 · 违反 → 流程中止）

| ID | 规则 | 防的失败模式 |
|----|------|-------------|
| **R1** | **证据紧跟**：正文每个定量断言同一句内有可点击 URL。无 URL = 删除断言 | 报告 100% 裸奔 |
| **R2** | **裸奔率闸门**：交付前程序化扫描；>10% 判 D 级退回；0% 通过 | 自评绕过 |
| **R3** | **多源验证**：定量锚点 ≥2 独立来源（不同原始数据源 + 不同利益集团）；单源率 >40% → "仅供方向参考" | 同源传播伪装共识 |
| **R4** | **原始数据保留**：每份研究员报告独立保存；缺 URL 的研究数据条目视为无效 | 证据链断裂 |
| **R5** | **假设可证伪 + 终审**：步骤一后形成 2-3 条可证伪假设；步骤六终审，被推翻的显式记录 | 调研变"罗列信息" |
| **R6** | **红队 ≥3 + 框架层**：含 ≥1 条攻击"分析框架本身的盲区"，指向具体数据点或推理链 | 红队变稻草人 |

#### Tier 2 · 防合规表演（7 条 · 违反 → 结论置信度降一级）

| ID | 规则 | 防的失败模式 |
|----|------|-------------|
| **R7** | **场景强制分解**：≥3 差异场景 + 需求权重 1-5 分量化 + 独立替代方案 | 把对象当整体讨论 |
| **R8** | **消亡案例**：≥1 个被放弃的方案/公司/思路；找不到显式声明"领域太新" | 连续上升叙事 |
| **R9** | **决策矩阵**：二维表（Top 3-5 方案 × 决策维度）+ 场景推荐规则 | "描述不判断" |
| **R10** | **论证强度底线**：最终结论 ≤ 最弱支撑链；弱证据禁用精确数字 | 弱证据撑强结论 |
| **R11** | **收敛自洽**：扫描全部结论，A 与非 A 不可同时出现 | 逻辑矛盾未被发现 |
| **R12** | **广度覆盖达标**：地理 ≥2 市场（含中国）+ 类型覆盖 + 覆盖率声明 | 单一市场/视角偏见 |
| **R18** | **范式深度拆解**：每个识别的主要技术范式（≥2）完成 (a)组件清单 (b)运行步骤 (c)优缺点矩阵 (d)同范式竞品对比；任一范式缺任一项 → 步骤标记未完成 | 技术分析漂浮在"黑盒"表面，只说"用了什么技术"但不说"怎么组成的、怎么跑的、为什么这么选" |

#### Tier 3 · 廉价高杠杆（3 条 · 违反 → 合规声明标注"未执行"及原因）

合规成本极低、信号价值极高。没理由不保留。

| ID | 规则 | 防的失败模式 |
|----|------|-------------|
| **R13** | **认知局限正文出现**：无知声明的核心不确定性在正文至少出现 1 次，不能完全隐藏到附录 | 把"不知道"藏起来，假装全知 |
| **R14** | **证据脆弱性 Top 3**：列出最脆弱的 3 个结论 + 最弱证据被推翻后的影响 | 报告看起来扎实但底部是空的 |
| **R15** | **数据时效标注**：每个定量数据点标 YYYY-MM-DD 截取日期；>3 月 benchmark 标"⚠️ 可能已过时"；>6 月权重降一级；禁用父类增长率推子类 | 过时数据伪装当前数据 |

#### Tier 4 · 元质量（2 条 · 触发自审重新评估）

| ID | 规则 | 防的失败模式 |
|----|------|-------------|
| **R16** | **质量终评**：6 维字母评级（独立性/交叉验证/证据金字塔/框架可比/工具边界/可执行性）+ "可用于 X / 不可用于 Y" + 最该补的 3 项；任一维 D → 总评上限 C；禁止自评全 A | 合规报告无质量判断 |
| **R17** | **方法论诚实自检**：每次执行后回答 3 问——(a) 哪些规则被形式满足但实质敷衍（列规则 ID）(b) 裸 prompt 跑 3 次取最好是否会更好 (c) 合规成本最贵的部分是哪 | 流程完美但产出低质 |

### 4.2 广度强化要求（R12 · 研究员强制）

| 要求 | 具体内容 | 在报告中的位置 |
|------|---------|--------------|
| **地理覆盖** | 必须包含中国市场分析；覆盖 ≥2 个地理市场（如中国 + 北美 + 欧洲） | 步骤一和步骤四中显式标注 |
| **类型覆盖** | 覆盖开源/闭源、商业/社区、成熟/新兴等不同类型 | 竞品矩阵中标注产品类型标签 |
| **覆盖率声明** | 显式声明"本报告覆盖已知 Top N 产品中的 X/N"，含遗漏清单 | 报告末尾"覆盖度声明"章节 |

### 4.3 执行方式

**串行启动**：N 个研究员 Agent 逐个启动，使用完全相同的 prompt。每个研究员完成全部调研并输出报告后，再启动下一个。

**<critical> 强制规则：所有研究员的 prompt 必须完全一致。</critical>** 只替换 `[对象名称]` 和 `[范围界定]` 两个占位符。严禁为不同研究员添加不同搜索角度、关键词建议或维度引导。

**串行目的**：N 个 Agent 面对同一份全维度指令，因搜索路径、关键词选择、信息来源的随机差异而产生不同发现——这是交叉验证的统计基础，不需要也不应该刻意引导。串行执行避免了并行工具的限流冲突，每个研究员可以拿到前一个研究员未触及的盲区方向作为隐性参照。

**关键警告——同模型 ≠ 独立验证**：

如果所有研究员使用同一底层模型，它们的"独立发现"不是真正独立的。共享同一知识盲区、训练数据截断日期、认知偏见。必须采取以下措施：

1. 交叉验证表中标注每个研究员的底层模型，格式 `[Agent N — 模型名称]`
2. 同模型多 Agent 的一致发现标注为"同模型复现"，不视为多源独立验证
3. 如果所有 Agent 使用同一模型，结论置信度上限为"中"

### 4.4 研究员 Agent 指令模板

```
## 8 步方法论

每个步骤末尾的交叉分析采用统一结构：**内部张力 → 跨步矛盾 → 论证链检验**。论证强度必须显式填写。

### 一、产品/场景调研

先搞清楚"是什么、谁在用、怎么用"，再谈技术。

**<critical> 深度优先于广度原则</critical>**：
- 如果覆盖对象 ≤ 5 个，每个必须有独立来源的详细分析
- 如果覆盖对象 > 5 个，允许以表格形式呈现边缘对象，但 **Top 3 必须有深入的独立分析**
- "深入分析"的可操作定义——必须同时包含三项：
  - (a) ≥1 个独立来源的评价（非官方文档/非厂商自报数据）
  - (b) ≥1 个具体使用案例或用户真实体验（来自论坛/评测/社交媒体，标注来源）
  - (c) ≥1 个该产品/方案已知的局限或失败案例
  - 缺少任一项 → 不算深入分析

**<critical> 场景强制分解（R7 · 强制）</critical>**：
- ≥3 个差异显著的场景，每个场景有独立的需求特征（准确率/速度/成本/可追溯性，1-5 分量化）+ 竞品替代方案 + 场景间互斥分析
- 无法拆解出 3 个 → 显式声明原因

- **产品形态**：面向谁、解决什么问题、核心交互
- **地理覆盖（R12）**：标注每个产品/方案的活跃地理市场，**必须包含中国**

**交叉分析**：
- **内部张力**：场景两两之间的需求冲突矩阵——场景 A 对性能的极致要求和场景 B 对成本的极致要求是否互斥？
- **跨步矛盾**：产品宣称解决的场景 vs 步骤二中揭示的实际技术能力是否匹配？
- **论证链检验**：如果核心用户的需求权重发生变化，整个产品形态是否仍然成立？论证强度：[待评估]

### 核心假设（步骤一之后必须输出 · R5）

基于场景调研，形成 **2-3 条可被证伪的假设**（格式：`如果 X 成立，那么我们应该看到 Y`）。假设必须是：
1. **可被证伪的**——后续步骤的信息可以推翻它
2. **高影响力的**——如果成立/不成立，会显著改变最终结论
3. **有可观测预测的**——"我们应该看到 Y"是可查证的具体现象

**<critical> 假设置信度锁定</critical>**：假设的置信度在首次评估后只能因新证据而改变。如果某步骤的假设检查返回"0（信息不足）"，终检时不得升级为"部分验证"或"已验证"，除非该步骤之后有新搜索/新来源直接提供了之前缺失的信息。无新证据的升级 → 置信度通胀，假设终检结果强制降为"🔴 未验证"。

### 二、技术范式/原理/实现

搞清楚"怎么做到的"。

- **核心原理**：关键技术是什么，依赖什么底层能力
- **架构骨架**：方案反复出现的结构模式
- **关键实现细节**：对性能/效果有决定性影响的具体设计选择
- **取舍结构**：共通的取舍（精确 vs 模糊、延迟 vs 吞吐、启动成本 vs 稳态性能）

**<critical> 范式深度拆解（R18 · 强制）</critical>**：

对调研对象涉及的每个主要技术范式（≥2 个），必须逐范式完成以下四项拆解。这是步骤二"深度"的核心保障——不做则技术分析漂浮在"黑盒"表面。

> **复用的已有技法**：本节的 (a) 对应附录 B "第一性原理拆解"；(c) 对应"失效模式分类"；(d) 对应"神帝思考"。已掌握这些技法的研究员可自然衔接。

**(a) 组件清单**

列出该范式的所有核心组件，以表格呈现：

| 组件 | 功能 | 输入 | 输出 | 是否可替换 | 替代方案（如可替换） | 依赖关系 |
|------|------|------|------|-----------|-------------------|---------|
| [组件名] | [一句话] | [输入数据/信号] | [输出数据/信号] | 是/否 | [替代组件] | 依赖 [X] 的输出 / 独立 |

- 组件间依赖关系总结：画出关键依赖链（如 A → B → C，其中 B 是瓶颈）
- 区分"范式必需组件"（所有实现都必须有）vs"可选增强组件"（部分实现选择性加入）

**(b) 运行步骤/流程**

按时间顺序列出该范式的完整运行流程（5-10 步），每步标注：

```

  输入：[上一步输出 + 外部输入]
  处理逻辑：[核心转换，1-2 句]
  输出：[产出物]
  决策点：[如有分支，列出分支条件和走向]
  瓶颈标注：🔴 延迟瓶颈 / 🟡 成本瓶颈 / — 非瓶颈
```

- 标注端到端延迟的主导因素（哪个步骤贡献 >30% 延迟）
- 标注不可并行化的串行段（如有）

**(c) 优缺点矩阵**

≥3 条优势 + ≥3 条劣势，每一项附带条件和分类：

| 优势 | 成立条件 | 劣势 | 触发条件 | 固有/阶段 |
|------|---------|------|---------|----------|
| [优势描述] | 在 [条件] 下成立 | [劣势描述] | 当 [情况] 时暴露 | 固有 / 阶段 |

- **固有劣势**：该范式的架构/数学模型决定的，即使技术进步也无法消除
- **阶段劣势**：当前技术/工程水平导致的，可能随时间改善（标注预计改善时间：如"1-2 年"）

**(d) 同范式竞品对比**

选定该范式下的 Top 3 实现方案/产品，对比关键差异：

| 方案 | 组件差异 | 步骤差异 | 关键设计选择 | 性能差异 | 适用场景 |
|------|---------|---------|-------------|---------|---------|
| [方案 A] | vs 基准 | vs 基准 | [最关键的 1 个不同选择] | [量化对比] | [场景] |

对比后回答两个核心问题：
1. **为什么不同**：同范式下方案 A 和方案 B 为什么做出不同设计选择？（场景定位不同？技术约束不同？历史路径不同？）
2. **不变核心 vs 可替换外围**：此范式的"不变核心"是什么（所有实现都必须做的事，3-5 项），"可替换外围"是什么（各实现自由选择的部分，3-5 项）？

```

  N1: [所有方案都必须做的事]
  N2: ...

  P2: ...
```

**<critical> 决策矩阵（R9 · 强制）</critical>**：
- 列 = Top 3-5 方案/产品
- 行 = 决策维度（准确性/速度/成本/生态兼容/可干预性等）
- 每格标注：✅ 优势 / ⚠️ 可接受 / ❌ 短板 / — 无数据
- 底部场景推荐规则：如果最关心 X，选 A；禁用"取决于"模糊表述

**交叉分析**：
- **内部张力**：架构骨架中的取舍是否存在零和博弈？
- **跨步矛盾**：技术选型是否和步骤一的场景需求自洽？
- **论证链检验**：如果底层依赖技术发生代际变化，当前取舍是否仍最优？论证强度：中

### 三、发展历程与时间线

搞清楚"怎么走到今天的"。按年份列出改变了走向的关键事件，不写流水账。

- 多线并行的对象**按线分述，不强行串成一条时间线**
- **<critical> 消亡案例（R8 · 强制）</critical>**：≥1 个被放弃的方案/公司/思路。找不到 → 显式声明"领域太新尚无完整消亡案例"
- **死而复生追问**：什么出现过、死了、后来又回来了？上次为什么死、这次为什么活？这一步防止进化论偏见——后出现的未必更好
- **路径依赖分析**：当前形态有多少是必然的、多少是历史偶然？哪些设计是"因为一开始那么做了，后面就改不了"？

**交叉分析**：
- **内部张力**：当前形态多少是必然（技术演进终点）vs 偶然（随机选择被锁定）？
- **跨步矛盾**：关键拐点与步骤四中竞品同期选择——为什么同一时间不同方案做了不同决定？
- **论证链检验**：把 2-3 个关键事件重新排序，终点是否会不同？会 → 格局脆弱；不会 → 深层结构性原因。论证强度：弱（反事实推理）

### 四、横向比较

并排对比，找差异的本质原因。

- **竞品发现**：≥3 个不同来源搜索（技术社区、学术搜索、产品对比站）+ 替代方案搜索（不依赖当前范式的替代方案？相邻领域产品？）
- **竞品矩阵 + 同期分岔点 + 关联技术演进**
- **类型覆盖（R12）**：标注产品类型标签（开源/闭源、商业/社区、成熟/新兴）

**<critical> 竞品 CEO 攻击表（建议）</critical>**：切换角色为竞品 CEO，攻击调研对象的核心优势：

| 调研对象声称的优势 | 竞品会如何攻击 | 攻击是否成立 | 如果成立影响多大 |
|------------------|--------------|-------------|----------------|
| [优势 1] | "这个优势其实是因为 [不可持续的原因]，在 [条件] 下会消失" | 成立/部分成立/不成立 | [评估] |

**<critical> 一手体验要求</critical>**：API 可调用/开源可运行 → 实际试用 Top 3 竞品，记录一手体验。无法实测 → 标注"未实测"，结论中降低对此产品的推荐置信度。

**交叉分析**：
- **内部张力**：竞品优势是否互斥？A 的优势恰恰是 B 的劣势？
- **跨步矛盾**：竞品矩阵 vs 步骤一场景拆解——不同场景下排名是否会翻转？
- **论证链检验**：竞争优势是结构性的（护城河）还是阶段性的（窗口期）？论证强度：弱

### 五、泛化分析

跳出固有范式，看更大的解空间。

- **范式外推**：当前范式在什么别的领域也成立？
- **新组合**：和别的范式拼在一起能出什么新东西？
- **反向推演**：把核心假设反过来，会得到什么？有先例吗？
- **<critical> 沉默证据</critical>**：行业里哪些问题几乎没人讨论？是解决不了、不重要、还是一致认为已解决但实际上没解决？
- **更优解空间**：不考虑现有约束，理论上最好的方案长什么样？将差距拆为三类：(1) 物理约束（不可逾越）；(2) 工程约束（可被足够资源解决，标注预期时间）；(3) 商业/生态约束（可被策略改变）。不可把商业约束伪装成物理约束

**交叉分析**：
- **内部张力**：泛化得到的多个新方向是否自洽？两个有前途的方向是否互斥？
- **跨步矛盾**：最被看好的泛化方向是否和步骤三/四中已验证的"死路"有相似特征？
- **论证链检验**：范式迁移是最弱论证链——如果目标领域核心前提不成立，范式迁移还成立吗？论证强度：弱

### 六、收敛

发散够了，收网。

**<critical> 假设终检（R5 · 强制）</critical>**：综合前五步发现——2-3 条假设中哪些被反复强化？哪些被推翻？被推翻的显式记录"HX 被推翻，依据：[证据]"，不可无声跳过。

**<critical> 论证强度终检（R10 · 强制）</critical>**：逐条回检步骤一至五中论证强度——被强化(+1)、维持(0)、被削弱(-1)？最终结论论证强度不得高于最弱支撑链。

**<critical> 收敛自洽检查（R11 · 硬性）</critical>**：列出所有结论两两比对——是否有 A 和非 A 同时出现？矛盾要么证明边界条件不同，要么写入"待验证冲突"，禁止同时采信矛盾双方。

**<critical> 甄别检查（≥80 字 + 提及具体名称）</critical>**：本步骤最可能出错的一项发现是什么？必须识别"收敛优先级错误"——机会方向排序是否依赖未验证前提。内容可套用到多个步骤的泛泛之谈 → 无效甄别。

**<critical> 红队回跑（R6）</critical>**：红队攻击中凡标注"颠覆"或"框架层颠覆"的条目，必须跳回受影响的前序步骤重新分析。最多 2 轮。第 3 轮仍出现颠覆级攻击 → 停止回跑，在结论中显式标注"存在未消解的颠覆性反证 [X]"，受影响结论降级为"待验证"。

### 七、红队反证时刻（R6 · 强制）

对初步结论做最后一轮对抗压力测试。切换角色为批评者。

**最小输出（不可跳过）**：
- ≥3 条红队攻击，每条含：攻击目标（针对哪个结论/发现）、攻击内容（具体质疑 + 证据或逻辑）、严重程度（颠覆/削弱/需限缩边界）
- ≥1 条**框架层攻击**：攻击"这个分析框架本身的盲区是什么"
- 红队使用前六步中实际发现的信息作为攻击武器——不是说"可能不准确"

### 八、结论 + 行动建议

**结论分层输出（R10 · 论证强度底线）**：

- 🔴 **立即行动**（1-2 条）：近期需要决策的事项，论证强度必须 ≥"中"
- 🟡 **趋势观察**（1-2 条）：中长期趋势，现在开始准备但不必立即行动
- 🟢 **风险警示**（1-2 条）：需要盯住的领先指标——什么信号出现时须重新评估

每点是判断不是描述。必须回答：选不选它？什么场景下选？天花板在哪？什么时候该换？最容易被忽视的风险是什么？

**<critical> 回头对无知声明</critical>**：之前不确定的现在有答案了吗？没有就直接说没有。把无知声明的核心不确定性在结论中至少出现一次（R13）。

**<critical> 证据脆弱性（R14）</critical>**：列出最脆弱的 3 个结论 + 支撑它们的证据中最弱的一环 + 如果被推翻对整体结论的影响。

## 输出要求
- 输出完整的独立调研报告，按八步骤顺序呈现
- 所有数据点标注来源 URL + 数据截取日期 YYYY-MM-DD（R15）
- 报告格式：HTML，使用 deepresearch/deep-research-template.html 模板
- 在报告末尾包含：
  - 覆盖度声明（R12）：已知 Top N 覆盖 X/N + 遗漏清单
  - 认知局限（R13）：无知声明核心在正文至少出现 1 次
  - 证据脆弱性 Top 3（R14）
  - 来源清单表格（来源名称、URL、分级、可信度、利益相关、数据截取日期）
```

### 4.5 研究员报告保存与打回重做

- **首次执行时，先创建目录** `[对象]/`（Bash: `mkdir -p "[对象]"`）
- 每次 Agent 返回完整报告后立即保存，命名 `[对象]/[对象]_Agent[N]_report.html`
- 所有研究员的 prompt 完全一致，不添加差异化引导
- **子报告不得删除**：最终交付物是 N 份子报告 + 1 份精编主报告 + 1 份审计报告

**打回重做机制**（由主编在 Phase 2 触发）：
- 主编按 5.8 打回标准校验，触发条件 → 打回研究员重做
- 研究员用原 prompt + **打回清单**作为修正指令重做
- 重做报告标 `_redo[N]` 后缀（如 `[对象]/[对象]_Agent1_report_redo1.html`），原始失败报告保留
- 单研究员最多打回 3 次，超过 3 次 → 该研究员报告标记为"**无效**"，主编在最终报告中显式声明并排除
- 如 N 个研究员全部被标记"无效" → 主编必须显式声明"无可用研究员报告"，置信度上限强制为"低"

## 五、Phase 2：主编执行

主编对最终报告的质量负全责。**角色定位：法官 + 审计师 + 编辑**。

### 5.0 主编前置

在开始校验和编辑之前，主编必须先完成以下前置工作：

#### 5.0.1 汇总研究员无知声明

读取 N 份研究员报告中的无知声明，识别：

- **研究员共同不确定项**：所有研究员都标注为不确定的内容 → 主编必须显式声明"以下问题全体研究员均无法确认"
- **研究员认知分歧**：研究员 A 确定但研究员 B 不确定的内容 → 主编优先解决分歧
- **主编层级的无知声明**：主编自己不确定什么？（汇总了研究员报告之后，主编自己可能知道新的盲区——你看到了研究员都漏了什么，但你自己的盲区是什么？）

#### 5.0.2 主编能力边界声明

- 声明主编可以使用的工具及其限制
- 声明因工具限制无法完成的校验项（例如"无法程序化扫描裸奔率"）

#### 5.0.3 主编定量锚定

- 汇总 N 份研究员报告的定量锚定，识别数字冲突
- 对冲突数字标注仲裁优先级（优先采信一手来源 + 最新时效）

### 5.1 来源可靠性校验

逐份研究员报告检查，每份抽检 ≥5 条 URL：

| 检查项 | 方法 | 判定 |
|--------|------|------|
| URL 真实性 | 验证 URL 是否可访问、是否与声称来源一致 | ✅ / ❌ |
| 来源分级正确性 | 一手/二手/三手是否符合域名判定规则 | ✅ / ⚠️ |
| 数据时效标注 | 关键 benchmark 是否标注 YYYY-MM-DD | ✅ / ❌ |
| 交叉验证完备性 | 关键数据点是否 ≥2 独立来源 | ✅ / ⚠️ |

**不合格处理**：≥3 个伪造/占位符 URL → 打回重做；1-2 个可疑 URL → 标注警告。

### 5.2 方法论可靠性校验

逐份研究员报告对照 17 条规则检查：

1. 场景分解是否 ≥3 场景？（R7）
2. 是否形成可证伪假设并在步骤六终审？（R5）
3. 红队攻击是否 ≥3 条且含框架层攻击？（R6）
4. 是否有消亡案例或显式声明"领域太新"？（R8）
5. 是否有决策矩阵？（R9）
6. 是否覆盖 ≥2 地理市场（含中国）？（R12）
7. 是否有覆盖率声明？（R12）
8. 认知局限是否在正文出现？（R13）
9. 是否列出了证据脆弱性 Top 3？（R14）
10. 关键数据是否标注了 YYYY-MM-DD 截取日期？（R15）
11. 每个识别的技术范式是否完成组件清单+运行步骤+优劣势矩阵+同范式竞品对比？（R18）

### 5.3 交叉验证

**事实核对**：
- 同一事实被多个研究员独立发现 → 标注"多源共识"
- 只有一个研究员发现 → 标注"单源待验证 [Agent N]"
- 不同研究员数字矛盾 → 追溯原始来源，优先采信一手来源

**数据冲突仲裁**：
- 追溯每个冲突到原始数据源
- 标注仲裁依据（来源等级/时效/利益相关）
- 无法仲裁的标注"待验证冲突"

**交叉验证对照表**：

| 核心发现 | Agent 1 | Agent 2 | ... | Agent N | 结论 | 置信度 |
|---------|---------|---------|-----|---------|------|--------|
| 发现 1 | ✅ | ✅ | ✅ | — | 多源共识 | 🟢 |
| 发现 2 | ✅ | ❌ 矛盾 | — | ✅ | 2/3 共识，分歧根因为... | 🟡 |

### 5.4 共同盲区分析（R9 四问）

所有研究员达成共识 ≠ 置信度高。必须回答：

1. **没问的问题**：对比所有研究员的搜索路径——它们都没问什么？
2. **没查的角度**：有没有共同遗漏的竞品、数据源、反对观点？
3. **共享的默认前提**：所有研究员都默认了什么？如果错了，哪些结论会崩塌？
4. **分歧根源**：研究员之间分歧的根源——不同信息来源？不同推理逻辑？不同初始假设？

### 5.5 红队对抗（最终轮）

主编对合成后的结论做最后一轮红队攻击：

- ≥3 条红队攻击，指向具体数据点或推理链
- ≥1 条框架层攻击
- 攻击使用研究员报告中实际发现的信息作为武器

### 5.6 行文定稿

**结构决策**：根据汇报对象决定行文结构，不被步骤顺序束缚。结构只为两个目标服务：(1) 30 秒抓住核心判断；(2) 深入阅读感到扎实可信。

**行文标准强制**：
- 金字塔结构，结论前置
- MECE：章节互斥且穷尽
- 每个段落段首回答"所以呢？"
- 关键数字前置——"市场规模：$0.26B（2025）[来源](URL)"
- 消灭弱措辞——"可能/或许/在一定程度上/似乎/倾向于"
- 单句 ≤ 40 字（中文）
- 全文一个一句话叙事（15 秒可读完）

**输出"行文策略说明"**：报告开篇附一句话——本报告面向谁、用什么结构、为什么。

### 5.7 主编输出

精编主报告（HTML），命名 `[对象]/[对象]_report.html`。

### 5.8 主编打回标准（主编 → 研究员）

主编校验研究员报告时，触发以下任一条件即**打回重做**。单个研究员最多打回 3 次，超过 3 次 → 该研究员报告标记为"无效"，主编在最终报告中显式声明并排除。

#### 硬性打回条件（触发即打回，无需累计）

| # | 条件 | 判定方法 |
|---|------|---------|
| **H1** | 裸奔率 > 10% | 执行 R2 程序化扫描，定量断言裸奔率超阈值 |
| **H2** | ≥3 个伪造/占位符 URL | 随机抽检 5 条 URL，验证可访问性和域名一致性 |
| **H3** | 关键数据点 ≥5 处无来源标注 | 逐条核对市场份额、性能数字、用户数等定量断言 |
| **H4** | 三手来源系统伪装为一手（≥3 处） | 域名与声称来源对比——"声称 Google 官方，实际链到聚合站" |
| **H5** | 8 步方法论缺失整个步骤 | 逐步检查，缺步骤（非缺步骤内细节）即触发 |
| **H6** | 无红队攻击或红队全部为稻草人 | 红队攻击未指向具体数据点，全为模糊质疑 |
| **H7** | 完全缺失前置 | 无无知声明、无定量锚定、无范围界定（缺 ≥2 项即触发） |
| **H8** | 缺失地理覆盖（中国） | R12 广度要求未达标，报告中无中国市场相关内容 |

#### 软性打回条件（累计 ≥3 项触发打回）

| # | 条件 | 判定方法 |
|---|------|---------|
| **S1** | 1-2 个可疑 URL（不可访问/域名不匹配） | URL 真实性抽检 |
| **S2** | 部分数据点缺来源标注（2-4 处） | 定量断言来源检查 |
| **S3** | 时效性标注不完整（≥3 处缺 YYYY-MM-DD） | 数据截取日期检查 |
| **S4** | 场景分解 <3 个（R7） | 方法论完整性检查 |
| **S5** | 无消亡案例且未声明"领域太新"（R8） | 步骤三检查 |
| **S6** | 无决策矩阵（R9） | 步骤二检查 |
| **S7** | 无覆盖率声明（R12） | 报告末尾检查 |
| **S8** | 范式深度拆解不完整（识别 ≥2 范式但任一缺组件/步骤/优劣势/竞品对比，R18） | 步骤二检查 |

#### 打回流程

1. 主编输出**打回清单**：具体数据点/缺失步骤 + 打回条件编号（如"缺 H8：无中国市场覆盖"）
2. 研究员用原 prompt + **打回清单**作为修正指令重做（见 4.5 打回重做机制）
3. 主编再次执行 5.1-5.6 完整校验
4. 累计 3 次打回未通过 → 研究员报告标记"**无效**"并排除
5. 所有研究员均无效 → 主编显式声明"无可用研究员报告"，置信度上限强制"低"

## 六、Phase 3：审计执行

### 6.0 审计前置

审计 Agent 在收到主编报告后、开始审计前，必须先完成以下声明：

#### 6.0.1 审计无知声明

- 作为独立第三方，你对审计对象的背景了解有什么局限？
- 哪些审计维度你可能无法做出可靠判断？（例如"非本领域专家，无法深入审计技术实现的正确性"）

#### 6.0.2 审计能力边界声明

- 测试 WebSearch 和 WebFetch 是否可用
- 声明因工具限制无法覆盖的审计项
- 如果只支持单一底层模型 → 标注"审计 Agent 与主编同模型，审计独立性受限"

#### 6.0.3 审计范围界定

- 明确审计覆盖范围和抽样策略（例如"随机抽样 10 条定量断言"、"重点审计结论 1-3"）
- 声明审计不覆盖的内容（例如"不对原始代码进行审计"、"不对采访录音进行验证"）

### 6.1 审计独立性保障

- 审计 Agent 为**全新启动**的 Agent，无任何前期上下文
- 仅收到主编精编报告文件路径
- 不知晓研究员数量、主编自评、内部方法论
- 独立 WebSearch，使用与研究员/主编完全不同的关键词
- **复评独立性受限声明**：如只支持单一底层模型，审计 Agent 仍与主编同模型——此时必须在审计报告中标注"审计独立性受限"

### 6.2 审计清单

#### 维度 1：事实准确性审计（定量）

随机抽取 5-10 条定量断言，独立 WebSearch 逐条验证：

| 审计项 | 方法 | 记录 |
|--------|------|------|
| 数字准确性 | 独立搜索验证每一条断言 | 断言内容 → 报告来源 URL → 审计验证结果 → 判定 |
| 来源可访问性 | 检查报告中的来源 URL 是否可访问 | ✅ / ❌ |
| 时效性 | 关键数据是否在有效期内 | ✅ / ⚠️ |

判定标准：✅ 准确 / ⚠️ 近似准确 / ❌ 错误 / ❓ 无法验证

#### 维度 2：方法论与逻辑审计（定性）

1. 报告的推理链是否完整？是否存在"因此显然"式跳跃？
2. 是否把"相关"当作"因果"？
3. 是否把"专家说"当作"事实"？
4. 结论的论证强度标注是否合理？
5. 是否存在同时采信矛盾双方的情况？

#### 维度 3：覆盖度审计

1. 是否存在重大遗漏的竞品/方案？（独立搜索验证）
2. 是否存在被所有研究者忽略的反对观点？
3. 地理覆盖是否充分？中国市场分析是否到位？
4. 类型覆盖是否全面？

#### 维度 4：偏见审计

1. 来源是否存在未披露的利益相关？
2. 是否存在幸存者偏差（只看到成功的案例）？
3. 是否存在确认偏误（选择性引用有利证据）？
4. 是否存在西方/英语中心主义偏见？

### 6.3 审计破坏性测试

选报告中最强的一个结论，用三种方式尝试破坏：

1. **假设数据采集有系统性偏误**——在什么条件下这些数字会是错的？
2. **假设关键来源有未披露的利益冲突**——搜索来源作者/机构的资助方、商业关联
3. **假设因果方向反了**——如果果是因、因是果，证据链会怎么重排？

### 6.4 审计报告输出

审计 Agent 输出独立审计报告（HTML），命名 `[对象]/[对象]_audit.html`。结构如下：

```

- 审计对象：[主编报告文件路径]
```

### 6.5 审计打回标准（审计 → 主编）

审计完成后，触发以下任一条件即**打回主编重做**。最多打回 3 次，超过 3 次 → **任务失败**。

#### 硬性打回条件（触发即打回）

| # | 条件 | 判定方法 |
|---|------|---------|
| **AH1** | 事实准确率 < 70% | 随机抽样 10 条定量断言独立验证，错误率 >30% |
| **AH2** | 存在系统性方法论违规 | 主编报告中 ≥3 处违反 Tier 1 规则（R1-R6）未修正 |
| **AH3** | 关键结论被审计证伪 | 审计独立搜索发现报告核心结论与事实矛盾 |
| **AH4** | 主编报告裸奔率 > 5% | 审计执行 R2 程序化扫描（主编门槛比研究员更严） |

#### 软性打回条件（累计 ≥2 项触发打回）

| # | 条件 | 判定方法 |
|---|------|---------|
| **AS1** | 事实准确率 70%-90% | 抽样有 1-3 条错误或近似错误 |
| **AS2** | 重大遗漏：缺失 ≥2 个已知主要竞品 | 审计独立搜索发现报告未覆盖的关键竞品 |
| **AS3** | 偏见未披露：≥2 处利益相关未标注 | 来源追溯检查 |
| **AS4** | 逻辑跳跃：≥3 处"相关当因果"或"专家意见当事实" | 推理链逐条审查 |
| **AS5** | 论证强度标注 ≥3 处不合理 | 结论与支撑证据强度不匹配 |
| **AS6** | 主编未回应研究员报告的已知分歧 | 交叉验证表检查——研究员之间的矛盾未被仲裁 |

#### 打回流程

1. 审计输出**打回清单**：具体问题 + 打回条件编号 + 修复建议
2. 主编根据打回清单修正精编报告
3. 修正报告标 `_redo[N]` 后缀（如 `_report_redo1.html`），原始报告保留
4. 审计再次执行完整审计清单（6.2 + 6.3）
5. 累计 3 次打回未通过 → **任务失败**

#### 任务失败输出

任务失败时，输出 `[对象]/[对象]_report_FAILED.html`：

```
封面：
  标题：深度调研报告：[对象] — 审计不通过
  标注：⚠️ 本报告经 3 次审计未通过，不可用于决策
  
正文：
  ├── 主编最终版精编报告
  ├── 附录 A：研究员独立报告索引
  ├── 附录 B：3 轮审计报告（按轮次排列）
  │   ├── 第 1 轮审计报告 + 打回清单
  │   ├── 第 2 轮审计报告 + 打回清单
  │   └── 第 3 轮审计报告 + 打回清单（失败）
  └── 附录 C：失败原因总结
      └── 3 轮均未通过的核心问题是什么
```

## 八、论证方法论（贯穿全流程）

以下三个框架贯穿全流程，研究员和主编均需使用。

### 框架一：正反合辩证

```
正（Thesis）：论证命题成立的条件和证据
    ↓
反（Antithesis）：系统性寻找反例、反证、失效条件
    ↓
合（Synthesis）：不折中——明确"正"和"反"各自的边界条件
```

- "合"不是各打五十大板，而是**明确边界**
- 找不到有力的"反"，说明调研不够深

### 框架二：四维交叉矩阵

| | 技术 | 时间 | 竞品 | 范式 |
|---|---|---|---|---|
| **技术** | — | 技术演进是渐进还是突变？ | 竞品的技术差异是根本性的还是表面的？ | 技术突破会否颠覆现有范式？ |
| **时间** | 历史拐点是否由技术突破驱动？ | — | 同期竞品为什么选了不同路线？ | 范式切换的触发条件是什么？ |
| **竞品** | 竞品的优势是结构性的还是阶段性的？ | 先发优势还是后发优势？ | — | 竞品是否属于不同范式？ |
| **范式** | 范式有没有被新技术颠覆的可能？ | 范式生命周期的阶段？ | 跨范式竞争的结果如何？ | — |

### 框架三：论证强度评级

| 等级 | 标识 | 判定标准 |
|------|------|---------|
| **强** | 🟢 | 多源独立证据 + 反例被系统性排除 + 边界条件清晰 |
| **中** | 🟡 | 有证据但存在单一来源、或反例未被充分排除 |
| **弱** | 🟠 | 基于类比/外推/专家判断，缺少直接证据 |
| **未验证** | 🔴 | 猜测、假设、待确认 |

## 九、材料可信度校验（研究员使用）

### 来源分级

- **一手来源**：原始发布者的域名。厂商数据在厂商自有域名下。论文在学术平台。可信度最高。
- **二手来源**：技术博客、行业报告、第三方评测、聚合站转载。可信度中等，需交叉验证。
- **三手来源**：社交媒体、论坛讨论、个人博客。仅作线索追踪。
- **不可信来源**：无出处数据、匿名爆料。直接排除。

### 来源被中转降级规则

链接域名与声称来源不一致 → 标注"来源被中转"，自动降一级。

### 交叉验证规则

关键数据至少需要两个独立来源印证。"独立来源"需同时满足：(1) 不引用同一原始数据源；(2) 不属于同一利益集团；(3) 不是同一发布链的上下游。追溯链断裂 → 标注"单源"处理。

## 本方法论的已知局限

1. **成本**：N 份研究员报告 + 1 份主编报告 + 1 份审计报告 = N+2 份文件。agents=3 时产生 5 份文件，token 消耗约为 deep-research.md Mode B 的 1.5-2 倍。

2. **审计独立性的模型上限**：环境只支持单一底层模型时，审计 Agent 与研究员/主编共享认知盲区，审计结论不应视为真正独立。

3. **地理覆盖的执行质量**：研究员可能"提到中国"但深度不足。主编需在方法论校验中重点检查地理覆盖的实质性。

4. **串行执行时间**：多个研究员串行执行会增加总耗时。N=3 时总调研时间约为单研究员的 3 倍。如需加速，可在 agents 参数减小 N 值。

5. **上下文窗口天花板**：复杂调研对象的精编报告可能超出单次上下文。主编需在行文阶段精简附录引用（保留链接，减少全文引用）。

6. **方法论的自我指涉（仍未解决）**：本方法论用于调研外部对象，但方法论自身的有效性从未被本方法论调研过。我们没有用这套方法论做过"Deep Research 方法论本身的深度调研"来判断它是否优于更简单的方法——这是一个递归漏洞。

以下技法都是**好工具**，在出现具体矛盾或需要深入时按需选用。**它们不是必填的合规盒子**——没有对应矛盾时强行套用反而催生合规表演。

> **技法与 R18 的关系**：以下带 ★ 的技法直接服务于 R18 范式深度拆解的四项输出，但技法本身仍是可选的——研究员可以选择用这些技法达到 R18 的要求，也可以用其他方法。

| 技法 | 适用场景 | R18 贡献 | 输出格式 |
|------|---------|---------|---------|
| **★ 神帝思考** | 步骤二，跨多产品找共性 | → (d) 同范式竞品对比 | 跨 ≥3 产品+≥3 竞品找到不约而同的设计决策；输出"不变核心 N: [描述] — [N 方案均遵循 / X 是例外，原因]" |
| **★ 第一性原理拆解** | 步骤二，剥离到不可约简约束 | → (a) 组件清单 | 物理约束 / 数学约束 / 结构约束三类剥离；追问"如果物理约束被突破，哪部分会变？"；输出"范式必需组件 vs 可选增强组件" |
| **★ 失效模式分类** | 步骤二，理解范式边界 | → (c) 优缺点矩阵 | ≥3 已知失效模式 + 真实案例 + 触发条件 + 来源 URL；每条失效模式对应优缺点矩阵中一个劣势 |
| **范式谱系** | 步骤三，暴露路径依赖 | 父范式 / 兄弟范式 / 已灭绝近亲 / 跨界来源 |
| **"死而复生"有效性测试** | 步骤三，防进化论偏见 | 每个案例通过两问：(1) 真死了吗（被合并/收购继续运营 ≠ 真死）？(2) 死因是本次分析关心的那个原因吗？ |
| **术语分岔搜索** | 步骤四，拓宽竞品发现 | 2-3 组不同词汇描述同一问题分别搜索，覆盖 ≥2 种语言 |
| **黑马挖掘** | 步骤四，避免头部偏见 | 搜 Reddit/HN/GitHub 找低曝光高口碑产品；GitHub star 增速高但总数 <5k |

| **范式组合相容性矩阵** | 步骤五，多范式叠加分析 | 协同（1+1>2）/ 可叠加 / 拮抗 矩阵 |
| **钢人论证** | Phase 2 审查，压力测试结论 | 为每条结论构建反结论的最强证据链；格式：`反结论 / 钢人论证（3 条最强证据）/ 能否共存` |
| **数字敏感性压力测试** | Phase 2 审查，测结论稳健性 | 3 个关键定量论断 ×0.5/×2/×10；容错 <2x → 标"脆弱的"；容错 >10x → 稳健 |
